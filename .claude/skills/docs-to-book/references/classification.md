# 七部分分类法

把散乱文档重排成「一本可顺读的书」，核心动作是**换分类轴**：不要沿用 `docs/` 的原始目录（那通常是按文档类型——prd/epic/guide/arch——或按时间堆出来的），要按**读者理解一个项目的自然认知顺序**重排。

## 认知曲线

读者要吃透一个陌生项目，理解路径是固定的：

```
这项目是干嘛的 ─→ 怎么长成这样的 ─→ 用什么方法论 ─→ 具体怎么实现
   (是什么)         (演进史)           (做事套路)         (落地细节)
   │                  │                  │                  │
   └─ 怎么跑起来 ─→ 数据在哪/怎么流通 ─→ 哪些规范要遵守
       (运维)           (数据层)             (约束)
```

落到七个部分（顺序即阅读顺序）：

| 序 | 部分 | 回答的问题 | 典型内容来源 |
|---|---|---|---|
| 0 | 起点导读 | 这书怎么读 / 项目一句话定位 | 综合 + 自撰 |
| 一 | 认识项目 | 是什么、架构概览、职责边界 | README、架构图、overview |
| 二 | 架构演进 | 怎么从 v1 长到现在、为何这么选 | epics、迁移记录、决策文档 |
| 三 | 方法论 | 做事的套路、领域知识、术语 | prd、设计文档、领域规则 |
| 四 | 业务实现 | 核心模块/工作流逐一拆解 | epic stories、功能文档 |
| 五 | 数据与集成 | 数据存在哪、怎么流通、外部依赖 | schema、数据源地图、集成文档 |
| 六 | 部署运维 | 怎么跑、怎么排查、怎么监控 | deploy、guide、ops 文档 |
| 七 | 参考规范 | 开发规范、技巧、漂移清单、未来 | rules、cheatsheet、TODO |

## 为什么要重排

原始 `docs/` 的目录通常长这样（按文档类型分）：

```
docs/
├── prd/          # 产品需求(但很多已实现)
├── epics/        # 史诗故事(状态滞后)
├── arch/         # 架构(和代码对不上)
├── guide/        # 操作指南(地址过时)
└── ...
```

这种目录**对写的人方便，对读的人灾难**——接手者想「这项目到底干嘛」得翻 prd，想「怎么跑起来」得翻 guide，想「数据在哪」得翻 arch，而这些文档互相重复、互相矛盾、状态不一。

重排成七部分后，读者**顺着一个轴走**：先知道是什么，再知道怎么来的，再懂方法论，再看实现细节……每一页都是上一页的自然延伸。

## 归类实操

通读后，给每个文档打两个标签：

- **它讲什么**（属于七部分中的哪部分）
- **它的可信度**（贴近代码现状 / 规划性质的 / 已过时）

然后：

1. 同一主题散落在多份文档的 → **合并**成一页，去重
2. 规划体「将要做 X」且已实现的 → 改陈述体并入对应页
3. 文档与代码矛盾的 → 不删，归入「漂移清单」页，标「以代码为准」
4. 纯历史参考、已被取代的 → 收进「EPIC 时间线」或脚注，不占主线

## 拆页粒度

- 每部分拆成若干页，每页一个明确主题，可独立读懂
- 单页别太长：若超过 ~300 行内容，考虑拆两页
- 用 slug 命名（kebab-case），如 `fed-rate`、`data-overview`、`drift-list`

## 确定全书骨架

归类完，落到 `book.config.js` 的结构里：

```js
parts: [
  { title: '起点导读', pages: ['index', 'reading-guide'] },
  { title: '一 · 认识项目', pages: ['overview', 'architecture', 'boundary'] },
  { title: '二 · 架构演进', pages: ['evolution', 'engine-choice', 'sandbox', 'orchestration'] },
  // ...
  { title: '七 · 参考规范', pages: ['rules', 'temporal-tips', 'epic-timeline', 'drift-list', 'scenarios'] },
]
```

`next/prev` 由构建脚本按这个顺序自动算，不用手写。

## 反模式

- ❌ 照搬 `docs/` 目录结构当导航（那是给写者的，不是给读者的）
- ❌ 一页塞一个原始文档全文（没有重叙述）
- ❌ 规划文档当现状讲（误导接手者）
- ❌ 把矛盾点悄悄删掉（应进漂移清单，透明告知）
- ❌ 部分太多（>8 部分读者记不住）或太少（<4 没分层）
