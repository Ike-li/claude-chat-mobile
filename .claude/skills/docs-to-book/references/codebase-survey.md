# 代码库探查 —— 无文档或文档残缺时的前置梳理

本 skill 默认项目有可读的 `docs/`。但现实里很多项目文档**缺失、残缺或严重过时**——这时不能空手做书,必须先让 agent 通读代码库,自己梳理出对项目的结构化理解,再进入归类与写书流程。

> 核心原则:**代码是唯一可信的事实来源**。文档会撒谎、会滞后、会自相矛盾;代码不会。当文档与代码冲突,永远以代码为准,并把冲突记入漂移清单。

## 何时需要代码探查

读完前置盘点后,按文档完备度走三条路径:

| 文档状态 | 判定信号 | 路径 |
|---|---|---|
| **完备** | `docs/` 存在、覆盖架构/数据/运维、内容不过时 | 以文档为主通读,代码仅用于核实漂移 |
| **残缺/过时** | `docs/` 存在但只覆盖局部,或大量「待办/待评审」、与代码明显不符 | 文档为线索,代码为主力:并行探查代码补全空白,交叉印证 |
| **无文档** | 几乎无 `docs/`,或只有零散 README/注释 | 完全以代码探查为核心,agent 自己从代码反推全书 |

下面三条路径都要落到同一份「全局笔记」,作为后续归类(七部分分类法)与写书的输入。

## 探查切入点（6 个角度，覆盖理解一个项目所需）

无论项目多大,从这 6 个角度切入,能拼出完整图景。每个角度都给子代理一个明确的「带什么回来」:

### 1. 入口与启动
- 主入口文件(`main.ts`/`index.js`/`app.py`/`Application` 启动类)
- 启动顺序:初始化什么、加载什么配置、起哪些进程/线程
- **带回来**:一张「系统怎么跑起来」的启动时序,标出各进程职责

### 2. 模块边界与依赖
- 顶层目录划分,每块负责什么
- 模块间依赖方向(谁是核心、谁调用谁、有没有禁止逆向依赖的约定)
- DI 容器/服务注册、依赖注入方式
- **带回来**:模块依赖图(Mermaid),标出「核心层」与「集成层」的边界

### 3. 数据流与状态
- 数据从哪进(Kafka/HTTP/Cron/WebSocket)、到哪去(数据库/消息/外部 API)
- 关键状态怎么流转(状态机、reducer pipeline、事件溯源)
- 核心业务对象的生命周期
- **带回来**:一条端到端数据流(Mermaid),从输入到输出

### 4. 持久化与数据层
- 用了哪些存储(MySQL/PG/Redis/ES/向量库/对象存储),各存什么
- schema 定义在哪(Prisma/Migration/实体类),关键表与字段
- 读写模式、幂等键、缓存策略
- **带回来**:数据存储全景表(实例→引擎→库→内容),标出哪个是「业务真相源」

### 5. 配置、部署与运维
- 部署形态(Docker Compose/K8s/裸机)、入口 compose/编排文件
- 环境变量清单(哪些必填、哪些易错配、默认值陷阱)
- 端口、队列、外部依赖地址
- **带回来**:一张「怎么本地跑起来」的最短路径 + 易错点清单

### 6. 外部集成与边界
- 依赖哪些外部系统(上游数据源、下游执行平台、第三方 API)
- 集成协议(HTTP/Kafka/gRPC)、鉴权、网络前提(VPN/白名单)
- 系统的**职责边界**:它做什么、不做什么(往往是最关键的认知)
- **带回来**:系统在生态中的位置图,明确「这事归谁」的边界

> 第 6 点的「职责边界」往往是接手者最需要、文档最缺的。务必从代码里把它问清楚:这个系统只生成信号不下单?只读不写?只管编排不管执行?——一句话能定义边界,胜过十页技术细节。

## 并行探查策略

代码量大(>2 万行)时,串行读会撑爆上下文。按**模块/目录**切分,并行派 Explore 子代理,各自只读一块、返回结构化摘要,主线再汇总。

```
# 切分维度(任选其一,按项目结构定)
- 按顶层目录:src/core、src/integration、src/persistence 各一个代理
- 按功能域:输入层、编排层、数据层、输出层 各一个代理
- 按子系统:每个业务工作流/模块一个代理

# 每个代理的统一输出格式（结构化,便于汇总）
- 这块负责什么（一句话）
- 关键文件与入口
- 对外依赖（依赖哪些模块/外部系统）
- 关键数据结构/表/状态
- 这块的「特殊约定或易错点」
- 文档与代码不一致处（如有）
```

委派约束(机制随环境,原则不变):
- 只读扫描优先用环境里的轻量/只读型子代理,降低成本;需要写文件才用通用代理
- 没有子代理机制时,主线按同样的切分串行探查,同样按上面的统一格式记笔记
- 无论何种机制:子代理只带回结构化摘要,禁止大段贴源码回来

## 从代码反推全书章节

探查产出的「全局笔记」要能直接喂给七部分分类法。代码事实到章节的映射:

| 代码里发现 | 归入书的哪部分 |
|---|---|
| 启动入口、模块依赖图、职责边界 | 一·认识项目(总览/架构/职责边界) |
| git log、废弃代码、`@deprecated`、新旧并存 | 二·架构演进 |
| 注释里的方法论、领域概念、分类体系 | 三·方法论 |
| 各业务工作流/模块的编排逻辑 | 四·业务实现 |
| 存储全景、schema、数据流 | 五·数据与集成 |
| compose、env、端口、运维脚本 | 六·部署运维 |
| 约定文件、规范、目录结构约束 | 七·参考规范 |

**演进逻辑怎么从代码挖**:看 git 历史(`git log --oneline | wc -l`、关键 merge commit)、看目录里有没有被取代的旧实现(如 `src/nodes/` vs `src/temporal/`)、看 schema 里有没有废弃表——这些是「项目怎么长成今天这样」的硬证据,文档常常不写。

## 文档-代码交叉印证

即使是「无文档」项目,也常有零散信息:README、CHANGELOG、代码注释、commit message、CI 配置、CLAUDE.md(如有)。把这些当**线索**,代码当**事实**:

1. 先从代码建立事实基线(模块图、数据流、职责边界)
2. 再拿零散文档去比对:哪里吻合(可信)、哪里冲突(记入漂移清单)、哪里文档有而代码没有(可能是规划未落地)
3. 冲突不删不改,单列漂移清单页,提示「以代码为准」

## 探查产物的沉淀

代码探查的「全局笔记」与通读文档的笔记**合并成一份**,统一作为归类与写书的输入。两者格式一致(跨文档/跨模块整合、演进逻辑、漂移点),主线不区分来源——读者最终看到的是一本连贯的书,而不是「文档部分」和「代码部分」的拼接。

## 常见坑

| 坑 | 解法 |
|---|---|
| 代理只读不总结,返回大段代码 | 要求「带回来结构化摘要」,禁止贴代码原文 |
| 切分粒度太细,代理互相重复 | 按不重叠的目录/模块切,明确各代理边界 |
| 拿注释当事实 | 注释是线索不是事实,以可执行代码为准 |
| 漏了职责边界 | 第 6 点必做,这是接手者最需要的认知 |
| 无文档就放弃演进章节 | 用 git log + 废弃代码反推演进,写进二·架构演进 |
