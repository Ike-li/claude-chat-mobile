# 阅读路线设计

读者打开这本「书」，应该立刻知道：**怎么读、读哪条线、按什么顺序**。这是用户明确要求的「详尽阅读顺序」，是核心交付物。

## 三条路线

给三类读者三条线，主线兜底：

| 路线 | 适合谁 | 时长 | 走法 |
|---|---|---|---|
| 主线通读 | 真要接手项目的人 | 2-3 小时 | 从第一页顺着 next 读到末页 |
| 速览 | 只想知道大概 | 20-30 分钟 | 只读各部分的「总览/概览」页 |
| 按角色切入 | 有明确职责 | 按需 | 直接跳对应部分，再回主线补 |

## 主线（必须存在）

主线 = 七部分的天然顺序，就是侧边栏从上到下、`next` 按钮一路点下去：

```
起点导读 → 一·认识 → 二·演进 → 三·方法论 → 四·实现 → 五·数据 → 六·运维 → 七·规范
```

这条线不需要额外设计，靠 `book.config.js` 的 parts/pages 顺序 + 上下页导航自动实现。但要在「阅读指南」页明确写出来，让读者知道顺着读就对。

## 速览线（建议有）

挑出每部分的「入口页/总览页」，串成 30 分钟能读完的最短路径。通常每部分第一页就是：

```
index → overview → evolution → methodology-summary → workflows-overview → data-overview → deploy-quickstart → drift-list
```

在阅读指南页用「30 分钟速览」小节列出，配跳转链接。

## 按角色线（建议有）

不同岗位关心的部分不同，给一条「按需切入」表：

| 角色 | 直接读 | 再回主线补 |
|---|---|---|
| 后端开发 | 四·实现 + 二·演进 | 三·方法论、七·规范 |
| 运维 / DevOps | 六·部署运维 + 五·数据 | 一·认识 |
| 量化 / 业务 | 三·方法论 + 四·实现 | 五·数据 |
| 产品 / 新人 | 一·认识 + 起点导读 | 全部 |
| 数据工程师 | 五·数据 + 四·实现 | 六·运维 |

角色表放在阅读指南页，读者对照自己的角色跳进去。

## 阅读指南页的结构

这一页是「书的地图」，建议结构：

```
# 阅读顺序

## 主线通读（2-3 小时）
<一句话 + 七部分顺序示意>

## 30 分钟速览
<挑出的 8 页 + 跳转链接>

## 按角色切入
<角色表>

## 几个约定
- 以代码为准 / 漂移清单在哪
- 暗色模式 / 搜索快捷键 等使用提示
```

## 阅读顺序的呈现

- **侧边栏**本身就是阅读顺序的可视化（按 parts 分组，pages 按序排）
- **上下页按钮**让顺读不断流
- **阅读指南页**用文字+示意图把「为什么这个顺序」讲清，给读者预期

## 反模式

- ❌ 只有侧边栏没有阅读指南页（读者不知道从哪开始、为什么这个序）
- ❌ 速览线跳过了「认识项目」（速览也得先知道是什么）
- ❌ 角色线列得太细（5 类够了，多了反而难选）
- ❌ 阅读顺序和侧边栏顺序不一致（会让读者困惑）
