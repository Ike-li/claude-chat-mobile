接手者最需要的认知往往是「这事归谁」。本节把职责边界写死，避免按 SaaS 或远程桌面去设计。

做什么

  • 把本机 claude CLI 接到手机浏览器（聊天 UI、工具卡片、审批、提问）。
  • 与终端共享同一会话 transcript（~/.claude/projects/…）与配置来源（user / project / local settings）。
  • 在 CLI 权限未命中时，把审批推到手机；命中则自动放行（与终端同源）。
  • 提供多工作区白名单、多会话标签、附件上传、PWA 与尽力而为的推送。
  • 提供 doctor / health / metrics 等运维面，便于自托管排查。

不做什么

不做说明
多租户 / 账号体系 / Hosted SaaS每实例单用户；任何通过鉴权的请求 = 坐在终端前的你
Claude 重实现不自带模型服务；不维护平行 agent 协议栈
项目级 allowedTools 放行表不注入 allowedTools / disallowedTools；放行集 = 用户已有 settings 的并集
无 token 的公网开放未设 AUTH_TOKEN 只绑 127.0.0.1
把 CLI transcript 迁入仓库控制面在 CCM_DATA_DIR；消息正文仍在 ~/.claude
跨实例联邦 / 远程管理 SaaS运维面有意克制

安全相关边界（摘要）

  1. 单用户。不要当多租户服务部署。
  2. 无 token 不出本机。
  3. 权限只继承 CLI。公网暴露前审查 ~/.claude/settings.json 白名单。
  4. 设备 TOFU。合法 token 不够；非本机设备须一次性批准。反代到 loopback 会跳过设备层——公网须 Access 或自觉强 token。
