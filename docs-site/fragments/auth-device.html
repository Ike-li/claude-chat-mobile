鉴权是纵深结构：token（或 Access JWT）证明「知道密钥」；设备 TOFU 证明「这台设备被主人批过」。合法 token alone 不足以操作未信任设备上的业务。

HTTP 与 Socket 共用

  • Token 匹配：HTTP 用头 x-auth-token 或查询 ?token=；Socket 用握手 auth。
  • 限速 Map 共享：/health、/metrics、/push/* 与 socket 握手共用，防无限试 token（默认连续失败阈值与锁定逻辑在 rate-limiter.js）。
  • 公网 Host 且 Access 启用时：优先 JWT；LAN / 其它 Host 用 token。
  • 无鉴权门（无 token 且非公网 Host）时限速不激活：本机调试路径 auth 恒过。

CF_ACCESS_HOSTNAME、CF_ACCESS_TEAM、CF_ACCESS_AUD 三项齐全才启用。校验经 jose + JWKS（缓存 cf-access-certs.json），失败则拒绝。任一缺失则整层关闭，回退 token 路径。改 env 后须重启常驻 server。

设备信赖（TOFU）

文件内容
$CCM_DATA_DIR/trusted-devices.json已信任设备（含 deviceToken）
$CCM_DATA_DIR/pending-devices.json等待批准的设备请求
  • 本机 loopback（127.0.0.1 / ::1 / localhost）可跳过设备门。
  • 非本机须批准后才能进入业务房间 approved。
  • CLI：node scripts/device.js list|approve <ID>|deny <ID>（与 server 读同一 CCM_DATA_DIR）。
  • 待批通过 pending_devices 等事件推到已信任端；UI 亦可批准。
危险任意把公网流量反代到 127.0.0.1 的拓扑（nginx/Caddy 的 proxy_pass localhost、SSH 转发、frp 等），都会让 peer 显「本机」，从而跳过设备审批，防线只剩 token（+ 若开 Access 则 JWT）。Cloudflare 命名隧道落到 localhost 时同样显本机，但公网有 Access 兜底，本配方可接受。

与审批台账的区别

设备 TOFU 管「谁可以连上来操作」；工具 permission_request 管「CLI 是否执行某条命令」。二者都是 fail-closed 纵深，不可互相替代。重启后设备信任仍在磁盘；在途工具审批回调死亡，磁盘 pending 会标过期。
