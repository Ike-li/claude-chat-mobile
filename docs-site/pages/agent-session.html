04 第四部分 · 核心实现

AgentSession

长驻 SDK query、权限闸、环形缓冲、排队与中断。

阅读约 14 分钟 第 13 / 28 篇 源自 src/agent/agent.js

AgentSession（src/agent/agent.js）是长驻 SDK query 的运行时核心：输入队列、权限闸、事件映射与环形缓冲。

职责

  • 持有一条到 Agent SDK 的 streaming input 会话（可 resume）。
  • 将 SDK 消息 map() 为契约事件并发出信封。
  • 通过 canUseTool 实现权限挂起/放行；维护 pending 审批与提问。
  • 支持排队、中断、撤回排队消息、切模型 / 权限档；effort 切档由 server 置换实例完成。

输入与忙碌

queue
streaming input 队列；busy 时可排队后续消息
pendingTurns
在途轮次计数
isBusy
pendingTurns > 0 或有后台任务或有 pending 审批/提问
idle hang
有 pendingTurns、无审批/提问且静默超过 idleTimeoutMs（默认 600000）则 abort

实例状态（instance-manager）：permission > busy > aborted/error/done > idle。

SDK 选项要点

  • cwd、pathToClaudeCodeExecutable、resume、permissionMode、effort
  • settingSources: ['user','project','local']
  • canUseTool 统一闸门
  • 不注入 allowedTools / disallowedTools

事件发出

  • emit(type, payload)：分配 seq，写入 cap=2000 的环形缓冲，回调 server onEvent。
  • emitTransient：不占 seq、不进缓冲（如 task_progress）。
  • 信封字段：seq, epoch, sessionId, instanceId, cwd, ts, type, payload。

权限闸 canUseTool

  1. 合法档六值：default / plan / acceptEdits / bypassPermissions / dontAsk / auto。
  2. dontAsk → deny（含 SDK 误调时的防御纵深）；bypassPermissions → 实例记真实 bypass，给 SDK 映射为 default，由 handleCanUseTool 直接 allow。
  3. AskUserQuestion 走 question 分支，不是普通 tool 审批卡片。
  4. 其余：算 fingerprint，写入 pending + 台账，emit permission_request，等待 user:approve。
  5. allow 时校验客户端回传 op 指纹，失败则 integrity_mismatch。
  6. TTL（approvalTtlMs，默认 1800000）到期自动 deny。
  7. 进程重启：磁盘 pending 全部 expired（回调已死），fail-closed。

排队、中断与撤回

  • busy 时再发可排队；system 提示「前面还有消息在排队…」。
  • user:interrupt：中断当前；kind interrupted；排队可能 queue_dropped。
  • user:cancelQueued：撤回排队中的消息；kind queue_cancelled，可带回 clientMessageId。
  • 勿靠中文文案字符串匹配终态，用 kind 字段。

模型与 effort

  • 模型可在发送路径切换；失败可 recoverable error 后用原模型发送。
  • effort 仅在 spawn 时注入；SDK 没有运行时 effort 控制请求。
  • Web user:setEffort：server 在可安全 dispose 时 silent dispose + dedupedResume 同 sessionId 新实例（带新 effort）；instanceId / epoch 变化；并发 setEffort 走 single-flight，避免双开 CLI。
  • null effort = 模型默认、不传 options.effort。
