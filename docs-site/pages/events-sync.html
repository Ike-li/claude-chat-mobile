事件信封与同步

agent:event 信封、seq/epoch、sync:since、catch-up。

阅读约 12 分钟 第 14 / 28 篇 源自 src/server/app.js · src/agent/agent.js

同步的单位是带序号的事件信封，而不是「整页 HTML」。客户端靠 seq / epoch 去重与补洞；磁盘侧另有 catch-up 追平 CLI 写入。

信封

{
  seq, epoch, sessionId, instanceId, cwd, ts,
  type,   // 契约内 agent:event type
  payload
}
  • seq：实例内单调递增；用于 sync:since。
  • epoch：实例世代；换实例（effort 的 dispose+resume、dirty 置换、reclaim 等）后变大，客户端必须重置 lastSeq 基线。
  • 入环形缓冲 cap=2000（BUFFER_CAP）；溢出丢最旧并可能标 gap。
  • emitTransient：不占 seq、不进缓冲（如 task_progress 心跳）。

广播策略

  • 默认发往已批准设备房间 approved（io.to('approved').emit('agent:event', …)）。
  • 高频 text_delta / thinking_delta：仅当该实例是当前 viewingInstanceId 时广播；后台 tab 仍写入缓冲，切回后用 sync:since 回放。
  • 边界事件驱动角标 latch、推送节流、会话活动 touch。
建议调试「后台会话像没字」时，先确认是否仅靠 live 流、是否切回后触发了 sync；后台不播高频 delta 是带宽设计，不是丢失。

视图守卫与 ack（2026-07-20）

  • server 对 send / cancelQueued 等写路径的 ack 须校验当前 viewing：切会话后不得把旧会话的 ack/错误刷到新 tab。
  • 前端 pendingSendBusy 等发送锁须按会话隔离，禁止跨会话污染（A 在忙导致 B 误显示忙碌）。
  • message-dedup：clientMessageId 幂等提交须防 TOCTOU（检查与 commit 之间并发重复发送）。
  • catchUpTickOnce：两分支（有会话 / 无会话）写全局 catchUp/mirror 状态时，须在视图守卫通过之后再提交，防止旧 tick 写脏新 viewing。

sync:since

  1. 客户端发送 { sessionId, lastSeq, instanceId }。
  2. 服务端 eventsSince(lastSeq)：过滤已 resolve 的审批/提问（避免重弹）。
  3. 若缓冲被 trim 且存在空洞 → 标记 gap 并提示。
  4. ack 可含 pending 快照（重建审批卡片）与 diskLen（对账终端写入量）。
  5. epoch 变化 = 新实例 → 客户端重置 lastSeq，不可混接旧流。
留意已 resolve 的审批不会从缓冲回放——必须以 pending 快照为准。极长轮次可能 gap，需依赖 history 全量或提示用户。

与 catch-up / mirror 的分工

机制数据源用途
环形缓冲 + sync:since进程内存信封Web 驾驶时的断线补发、切 tab 回放
catchUpTick / history_append磁盘 jsonlCLI 驾驶时只读追平；标 externalDirty
session:history磁盘 jsonl 冷读打开会话时全量/分页历史
