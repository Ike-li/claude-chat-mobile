单驾驶员模型

Web 与 CLI 不可同时写同一会话；镜像只读追平。

阅读约 10 分钟 第 9 / 28 篇 源自 src/agent/cli-mirror-state.js · src/sessions/history.js

同一 sessionId 在同一时刻只允许一个驾驶员写入。Web 与本机 CLI 不能并行写同一会话；后到的一方只读镜像或置换冷读，避免 transcript 分叉。

为什么需要单驾驶员

会话内容的权威落点是 CLI transcript（jsonl）。若 Web 侧 SDK 子进程与终端里正在跑的 CLI 同时向同一 sessionId 追加轮次，会出现双 parent 链、模型上下文滞后、手机与终端各说各话。单驾驶员把「谁可以 send」收成互斥问题，而不是事后合并冲突。

CLI 驾

Web 只读镜像

追平磁盘 transcript，输入锁为只读。

Web 驾

SDK 长驻 query

正常发送、审批、中断。

外部脏

externalDirty

磁盘已超前于 SDK 内存，发送前须置换。

三种形态

形态写入者Web 行为
Web 驾驶本进程 AgentSession正常发送；状态栏取 SDK
CLI 驾驶（只读镜像）终端 CLI输入锁定；catch-up 追 transcript；可选 statusline bridge
接管Web 在镜像下成功 send清 mirrorReadonly；dispose+resume 后 Web 驾驶

镜像机制

catchUpTick
轮询当前查看会话的 jsonl 增量 → history_append；只读时约 1s 一拍，常态约 2.5s
mirror_state
广播只读锁与观察态（model/permission 等）
mirrorEntryLock
切入会话时预锁（尾部 pending）
mirrorReleaseStep
安静约 12.5s 后自动释放只读锁（文件仍增长则 keepAlive）
cli-mirror-state
只读观察 CLI 落盘偏好，不写回 Web 实例偏好
forInstanceId 守卫
setMirror / catchUp 写入带 viewing 快照；切视图后旧 tick 不得改全局锁
clearMirrorOnViewChange
切会话/视图立即 force 清全局 mirror，防 A 会话锁套到 B（跨工作区误锁）
建议2026-07-20 修复点：镜像误重锁（解锁后不该被旧逻辑再次锁上）；catchUpTickOnce 两分支的状态提交延后到视图守卫之后，避免切 tab 后旧 tick 写脏全局 mirror / catchUp 状态。
附注Web resume 是冷读 transcript，不能 attach 正在运行的 CLI 活进程。正在终端里跑的会话，Web 侧只能镜像或等其结束后再 resume。
留意长工具若长时间不写盘，仅靠「文件增长」会误判静默；实现用 tailPending / keepAlive 等互补判据，避免横幅误灭成「感觉没在跑」。

externalDirty 与发送

  1. catch-up 或 rebaseline 发现磁盘已超前 → 标记 externalDirty。
  2. 用户发送时若实例 idle：dispose + resume，冷读磁盘后再发（吸收终端写入）。
  3. 若实例 busy：硬拒绝（NACK），可重试；不是静默排队。

externalDirty 只约束「下次 Web 写入前必须吸收磁盘」；它不单独把 statusline 锁到 CLI——statusline 是否走 CLI 快照看 mirrorReadonly。

flowchart LR A["查看会话"] --> B{"尾部/锁判定"} B -->|"CLI 在写"| C["mirrorReadonly"] C --> D["catchUpTick 追平"] D --> E["mirrorReleaseStep ~12.5s"] E -->|"静默满"| F["解锁可输入"] B -->|"外部已超前"| G["externalDirty"] G --> H{"发送时 idle?"} H -->|是| I["dispose + resume 冷读"] H -->|否| J["NACK"]

切回运行中会话（busy reseed）

从其他标签切回仍在跑的会话时，前端按 server 权威 state 重种运行指示，避免「其实还在跑，UI 却像空闲」。权威来源是 instances 等载荷中的 state，而非本地 spinner 记忆。

已知边界

  • mirror 全局单值（非 per-连接）；切视图会清理，设计上未做细粒度 per-socket 锁（AD-5 有意保留）。
  • catchUp 主要盯当前查看会话；后台 tab 被外部写过、切入后首个 tick 前极速发送，边界上可能不经 dirty 置换（已知）。
  • busy→idle 窗口内部分终端写入可能被吸收而不标脏（有意取舍，防重复气泡）。
  • 环形事件缓冲 2000 与历史消息 2000 上限是不同轴（事件信封 vs transcript 条数）。
