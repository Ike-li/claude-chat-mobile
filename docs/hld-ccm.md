## 一、引言

### 1.1 目的与范围

本文档承接 `prd-ccm.md` v3.5(提交评审稿,尚待评审通过),面向技术评审与详细设计。覆盖组件划分、关键架构决策(含论证与备选)、数据模型、关键流程、部署;不下沉到类级详细设计,不含界面视觉。

### 1.2 设计原则(承接 v3)

终端等价(下界+上界)、透明性、单操作者/机主即 root、断线重连是主路径、fail-closed、上游耦合收敛、**瘦中转(CC CLI 为唯一数据源)**。两条优先级约束:① **终端等价优先**——"通用最佳实践"与终端等价冲突时守终端等价(安全维度例外:限速/强认证不违反等价);② **瘦中转优先**——CC CLI 本身有的一律从 CLI 取、不自建平行数据源;自建仅限 CLI 没有且 web 必需者,并最小化。每多一个自建数据源即多一处漂移/同步/维护成本。

### 1.3 决策方法

每个架构决策(AD-n)给出:**需求依据**(v3 的 FR/NFR)→ **方案** → **论证与备选** → **与现有参考的关系**(采纳 / 改进 / 超越)。

## 二、架构概览(修正版)

> v1 的架构图有根本错误:它假设终端会话经过后端。真实架构中,**终端的 claude 是用户直接运行的独立进程,默认不经过后端**。这决定了同步与实时镜像的形态。

```text
┌── 移动客户端 ──┐        ┌────────────── 主机(操作者自有)──────────────┐
│  Web / PWA     │◀─TLS──▶│ 边缘接入层 ─▶ 后端服务 ─(Agent SDK)─▶ claude │
│  (浏览器)       │◀Push─  │   (反代/隧道)     │        (web 侧会话)         │
└────────────────┘        │                   └─▶ 元数据/审计/设备(轻量存储)│
                          │                                                 │
                          │  会话消息事实源:claude 的会话记录(transcript) │
                          │  ┌─────────────────────────────────────────┐   │
                          │  │ 终端 claude(用户直接运行,独立进程)      │   │
                          │  │   —— 默认不经后端;web 侧只冷读其记录     │   │
                          │  └─────────────────────────────────────────┘   │
                          └─────────────────────────────────────────────────┘
```

后端七组件:CLI 桥接层、会话管理器、事件流引擎、控制通道处理器、鉴权与设备信任、通知服务、可观测性(职责见第五章)。

## 三、关键架构决策

### AD-1 会话消息事实源与存储 [承接 FR-11/19/20、NFR-16]

- **方案。** 会话消息的事实源采用 **claude 自己的会话记录(transcript)**;后端**不另建消息库**。后端另维护**轻量存储**(会话元数据指针、审计事件、设备信任、推送订阅),用于审计查询(FR-19)与恢复。
- **论证。** transcript 为事实源使 web 与终端**天然共享同一会话、双向可续、不分叉**(直接满足 FR-11);另建消息库会制造第二事实源(v1 SQLite 方案的错误)。
- **瘦中转的落实。** 能从 CLI transcript 派生的一律不自存:**会话标题从首条消息派生**、**权限档从 transcript 记录读取**;CLI 未落盘者(如 **effort 档**)**不自存、不恢复**(接受 CLI 默认档——零自建换零复杂度);后端仅保留 CLI 根本没有的最小状态(当前指针、设备信任、推送订阅、最小审计)。审计**不拔高**为独立查询系统,仅记 web 特有安全动作的元数据。
- **删除边界(FR-20 落地,Could/纯数据主权)。** 定义两级:①删除**本产品可见的会话引用**(默认);②可选删除**底层 transcript 文件**(显式二次确认)。满足"删除暴露到服务上的数据"这一主权诉求(非终端等价——见 v3 §12.2),诚实处理"事实源属 claude"的边界(对应 v3 OQ-06)。
- **读取契约(官方文档校准)。** transcript 的**文件路径**是官方稳定契约(hook 输入 `transcript_path`),但 **JSONL 内部格式(字段/结构)未文档化、无稳定承诺**;故消息读取**走官方公开的 session-messages 读取 API**(具体签名以官方 SDK 文档为准、实现时核定),**不自行解析 JSONL 格式**——只把"路径"当契约,规避格式随版本漂移。这是独立查证官方文档(而非"现有能跑")得出的校准。读取一律走该 API 的分页(limit/offset,SP-07 已坐实),历史回放与完整重载不整包拉取,承接 PRD"小屏弱网"前提。
- **与现有关系。** **采纳**"transcript 事实源 + 只存元数据"(独立判断其为双向共享的最简正解,也是瘦中转的典范);仅在**数据删除**上**改进**(现有不做);审计**保持最小**、标题**改为派生**、effort **不自存**(比现有更瘦);消息读取走官方 API(**改进**:不绑定 JSONL 格式)。

### AD-2 会话共享与接续 [承接 FR-11]

- **方案。** 会话归属以"其记录存在于对应工作目录"为准,故终端建的会话与 web 建的会话彼此可见、可续。web 接续通过驱动一个 `resume` 会话读取既有记录。
- **与现有关系。** **采纳**(现有以此实现双向互见,经测试验证为正确良构)。

### AD-3 实时镜像活跃会话 [承接 FR-13a 下界(Must) / FR-13b 理想目标]

**需求依据。** FR-13a(下界,Must):web 只读呈现活跃会话**已落定**的内容与增量进展;FR-13b(理想目标):进一步看到与终端等价的实时进展(含进行中 thinking/子任务)。**约束**:终端等价优先 + **瘦中转**——不为此自建重型机制。

**问题根因。** 实时进展存在于**驱动 claude 的那个进程的内存**,进行中态不落盘;web 冷读磁盘只能看到已落定内容。障碍是"驱动 SDK 的进程不是后端"——而这**本质是 CC CLI / SDK 是否提供旁观活进程能力的问题,应由上游解决,不该 web 自建数据源去补**。

**备选方案评估:**

| 方案 | 实时性 | 不偏离原生 claude | 自建复杂度 | 裁决 |
| --- | --- | --- | --- | --- |
| A 后端成为唯一驱动者 | ✔ 全实时 | ✘ 违反(不再用原生 claude) | 高 | 否决 |
| B **依赖上游 attach 能力** | ✔ | ✔ | **零(从 CLI 拿)** | **首选方向(等上游)** |
| C 协同驱动(co-driven) | ✔ 该会话实时 | ◑ | 高 | **否决(现有技术不可行:需上游未提供的 attach;且偏离原生 claude)** |
| D 冷读磁盘 | 仅已落定(非实时) | ✔ | 低 | **v1 基线** |

**决策:**
- **v1 基线 = D(满足 FR-13a 下界)**:纯原生会话 web 只读呈现已落定内容,诚实告知"活跃会话进行中态不可见"。
- **FR-13a"增量进展"的追平通路(下界的动态化,独立于 AD-4)**:FR-13a 要的不止一次性快照——活跃会话(**含终端在旁写入**)新落盘的条目须能持续追平到已连接的 web。机制:客户端维护已渲染的 transcript 长度 `seenDiskLen`,重连或观察期对**当前单个会话**探测,比对后端 `diskLen`,增量下发新落定条目。**这条通路必须独立于 AD-4 的 server 事件流补齐**——终端写入不经后端、不产生 server `seq`,若只靠 seq 补齐则终端的外部写入对 web 永久不可见(结构性丢失)。仅针对当前观察会话、不轮询所有会话,不违反 EP-1。算法落 LLD §5.1。两点架构级约束:①一致性——web 本地驱动产生的消息同样落盘推高 diskLen,故 seenDiskLen 须随本地已渲染事件同步推进,事件流补齐(AD-4)与 diskLen 追平两路增量的去重与次序一致性是 LLD §5.1 必须解决的约束,非可选细节;②diskLen/seenDiskLen 以读取 API 返回的消息条数计,不以 transcript 文件行/字节计(守 AD-1"路径为契约、格式无承诺")。
- **首选缩小差距 = B(依赖上游)**:实时态本应由 CC CLI / SDK 提供(attach / 旁观活进程)。**优先等待并采用上游能力,而非 web 自建**;登记为上游依赖并持续追踪(SP-06)。
- **C(co-driven)已否决**:自建终端侧驱动器 + 双路渲染需 attach 活进程能力(上游未提供)——**现有技术下不可行**,且偏离"用户用原生 claude"、维护复杂度高。它因**技术不可行**被排除,非因价值排序被牺牲,故不构成对"终端等价优先"(§1.2)的违反。
- **不接受现状为需求上限**:FR-13b 作为理想目标保留;缩小差距的正当路径是**推动/等待上游**,不是 web 自建。

**与现有关系。** **暂同现有(接受冷读下界)+ 依赖上游演进**。较早稿曾评估自建 co-driven,因现有技术不可行且偏离原生 claude 已否决;实时性依赖上游 attach 能力(SP-06),本期不承诺。

### AD-4 事件流与断线补齐 [承接 FR-02/03、NFR-10/14]

- **方案。** 统一事件信封承载 server→client 全部事件;信封含**进展来源标识**(`origin`:终端 / 移动端,承接 FR-02——操作者须能区分某进展是自己在移动端发起还是终端产生);每会话单调序号 + 每实例跨重启唯一标识(供客户端重置去重基线);有限缓冲供断线增量补齐,超窗口回退完整重载;高频心跳走非补齐旁路避免挤占缓冲;事件类型集受可执行契约守卫防漂移(real ⊇ mock)。
- **与现有关系。** **采纳**(独立判断此为良构;v1 独立稿亦殊途同归推出信封+序号骨架,现有补足了跨重启标识与心跳旁路两个我曾遗漏的要点)。

### AD-5 多入口并发保护 [承接 FR-12]

- **方案。** 保证单一有效写入者;检测到某入口不能安全写入时,将其**置为只读并告知用户**,防两进程写同一记录致分叉;提供清晰的**接管路径**缓解体验损耗;经静默判据自动解除只读。
- **改进点(对现有已知缺陷)。** 现有的只读状态为**全局广播 + 单一活跃视图**,导致"两设备看不同会话时一方解锁误解另一方"(现有有意不修)。本稿**改进**为 **per-会话 / per-连接的只读状态 + 定向下发**,消除跨设备误解——这是现有自身承认"完整修复需要的方向"。
- **与现有关系。** **采纳**只读保护机制(经测试验证有效)+ **改进**其跨设备缺陷。

### AD-6 核心引擎:CLI 驱动 + 桥接映射层 [承接 NFR-14]

- **驱动(输入侧)。** 用 **Agent SDK 的 `query()`** 驱动**本机 claude**(`pathToClaudeCodeExecutable` 指向本机可执行文件,而非 SDK 捆绑副本);采用**流式输入**(streaming input——持续 yield 用户消息,以支持中断与 `canUseTool`)+ **流式输出**(`includePartialMessages`)。**关键**:后端**不自己 `spawn` 子进程、也不直接操作 stream-json 协议**——启动 claude 子进程与 stream-json 传输均由 **SDK 代劳**,后端只调 `query()`。这是瘦中转在引擎层的体现:连进程启动与协议解析都不自建。
- **映射(输出侧)。** SDK 吐出的消息经**唯一一处映射层**翻译为契约事件——唯一的上游耦合点;处理子任务流过滤、批量增量、模式切换兜底、提问特判等 SDK 真实行为;上游破坏性变更仅此处适配,并经契约回归守护。
- **与现有关系。** **采纳**(独立判断"用 SDK 驱动 + 集中映射"是收敛上游耦合的正解;v1 独立稿亦推出映射层结论)。

> **一句话核心引擎**:用 Agent SDK 的 `query()` 驱动本机 claude(流式输入 + 流式输出);`spawn` 子进程与 stream-json 协议由 SDK 代劳,后端只负责"喂输入消息 + 把 SDK 消息映射成前端事件"。—— 常见的"spawn 本机 claude + stream-json"说法漏了 SDK 这一层:那两件事是 SDK 干的,后端不直接做。

### AD-7 分层鉴权 + 防暴破 + 强认证 + 审批完整性 [承接 NFR-01/02/03/04/17 —— 含新增]

- **分层鉴权(采纳)。** 边缘接入层(反代/隧道/身份代理,fail-closed)+ 应用层,两层独立、按 Host 二选一而非叠加;凭证只从启动 shell 注入、不下发移动端、配置文件最小权限原子写。**采纳**现有的分层与凭证隔离(独立判断其良构)。
- **防暴破限速(新增,NFR-03)。** 鉴权端点须有失败限速 / 退避 / 锁定 / 告警。现有**无**此机制(判为缺口)。架构:对失败尝试按来源计数、指数退避、阈值锁定并计入审计与告警。**超越**现有。
- **强设备认证(新增,NFR-04)。** 演进为设备绑定令牌 + **WebAuthn/passkey** 优先,替代"可转发、撤销粗粒度的静态共享口令为唯一手段";保留静态令牌作为降级/首配路径。**超越**现有。
- **审批完整性(新增,NFR-17)。** 移动端展示待批的操作,须与主机实际执行的操作**绑定一致**——审批请求承载操作指纹(命令+参数+cwd 的完整摘要),执行前校验"所批即所行",展示与执行不一致则 fail-closed 拒绝。防中间层在展示与执行之间篡改。现有**无**此绑定校验(判为缺口)。**超越**现有。
- **重启后 pending 审批的 fail-closed 处置(新增,承接 NFR-09/11)。** 审批挂起依赖活的 `canUseTool` 回调(存于驱动进程内存);后端重启则该回调随进程终止、挂起的审批无法再兑现。故重启后,持久化表中 `status=pending` 的审批一律**标记失效**(而非显示"仍可批准"),提示操作者重新触发——执行上下文不可确认时默认拒绝。防"重启后批准一个不会被执行的操作"的假象。

### AD-8 设备信任 [承接 FR-18]

- **方案。** 首次接入显式授权的信任模型;多路审批(终端/命令行/已信任设备远程)、信任变更即时生效与自动解锁、待审队列上限防滥用、审批权恒属已信任设备。
- **与现有关系。** **采纳**(经测试验证的良构);与 AD-7 强认证叠加增强。

### AD-9 通知 [承接 FR-14/15]

- **方案。** 后台推送(标准 Web Push + 自签密钥)+ **备用推送通道**绕过特定平台后台限制;触发时机:仅需人工决策或任务终结时推送(中间过程不扰),含深链;不支持后台推送的平台以重连后状态兜底。通知正文策略(承接 PRD OQ-08):标准 Web Push payload 端到端加密(RFC 8291),push service 不见明文;备用推送通道无此标准保证,正文可能明文经第三方——故通知正文默认最小化(定位+深链,不含命令/路径等潜在敏感内容),具体脱敏边界落 LLD;push service 属第三方信任边界,架构图未单独绘出、于此登记。
- **与现有关系。** **采纳**(含备用通道的现实补充,独立判断其必要)。

### AD-10 任务状态派生 [承接 v3 第八章状态模型]

- **方案。** 后端由在途轮次、挂起审批/提问、错误/完成信号,派生 `运行中/等待审批/等待输入/已完成/已失败/已中止` 六态;**"主机离线"不可能由后端派生(后端存活即主机在线),它由客户端在与后端失联时于连接层本地派生**,后端仅提供心跳/连接信号供客户端判定。规则:"完成"必为真实终态;状态不可确认时客户端呈现"主机离线"而非假定完成。
- **与现有关系。** **采纳**派生方式(实现良构)+ **确保覆盖** v3 要求的完整状态集(现有派生 5 态,须补齐"已中止"等用户须区分的状态;"主机离线"划归客户端职责为本版澄清)。

### AD-11 会话发现与"等我"聚合 [承接 FR-21]

**需求依据。** FR-21(Must):操作者离席时本机可能有多个会话(多项目 / 终端与 web 各建),须能在移动端快速发现并定位"哪个在等我"(等待审批 / 输入),而非逐个点开排查。正当性 = **注意力不对称**(v3 §2),非终端等价。

**上界自洽(防"终端没有的不加"质疑)。** 分两层:①**基础会话列表**(有哪些会话、cwd / 标题 / 最近活跃)= 终端 `claude --resume` 选择器的语义等价(终端等价下界);②**"等我"聚合与紧迫度排序** = 终端选择器没有的增强,但它是移动端特有问题(注意力不对称)的对偶,与推送通知(AD-9)**同源正当**,不属"上界膨胀"。

**约束。** 瘦中转(EP-1:不为发现而周期轮询所有会话 transcript)+ 继承 **AD-3 边界**(终端活跃会话的进行中 / 等待态在其驱动进程内存、不落盘,后端冷读不可见)。

**方案(读模型投影,非新数据源)。**
- 数据源①(等我审批):后端持久化的 `approval_request`(status=pending 且未过期)——跨会话、跨重启可靠,不依赖进程内存。
- 数据源②(等我输入):对 web 后端在驱动的会话,由 StatusDeriver 纯函数派生 `awaiting_input` / `awaiting_approval`;复用 AD-9 TriggerPolicy 已识别的"需人工干预"时刻作为置位信号(仅观察、未驱动的会话无此数据源,其经后端拦截过的审批仍由数据源①覆盖——与下方边界一致)。
- **悬置起点(供紧迫度排序 + FR-22 悬置时长)的数据源。** "等了多久"需要一个**进入等待的起始时刻**,它**不能从过期时刻 `expiresAt` 反推**(TTL 在 OQ-05 尚未定)。故:审批维度取 `approval_request.createdAt`(审批被创建的时刻);等输入维度取会话运行时的 `awaitingSince`(进入 `awaiting_input` 时由状态转移置位的时间戳)。二者才是 `waitingSince` 的真实来源(字段见第四章数据模型,LLD 落到 schema)。
- 投影:SessionManager 增设纯函数 `deriveAttention(sessions, pendingApprovals) → AttentionView`,聚合 awaiting_* 会话并按紧迫度(悬置时长 / 风险 / 项目分组——权重见 **OQ-01**)排序。**即时派生、不落盘**(EP-1)。
- 前端 IA:会话列表页顶部"需要你(N)"区置顶聚合(深链单会话,复用 FR-14),其下全部会话(按 cwd 项目分组、`lastActiveAt` 排序)。

**聚合的更新与失效时机。** 纯函数即时派生,无需显式"移除"逻辑:审批被批准/拒绝/过期后,下次派生时数据源①的过滤条件(`status=pending 且未过期`)自然将其排除;等待项在会话推进/终结后由状态派生自然离开 awaiting_*。**触发重算**复用既有事件——`approval_request` 变更、`turn_end`、`input_needed`(AD-9 TriggerPolicy 已识别的信号)即驱动重新投影;多设备一致性由 AD-5 的 per-会话定向下发 + 事件去重承接,不另设机制。

**边界(诚实登记,继承 AD-3)。** 聚合的**可靠覆盖 = 仅 web 后端拦截过 / 驱动过的会话**:①"等我审批"仅在审批经后端 `PermissionInterceptor`(有 `approval_request` 记录)时可聚合——**纯终端会话的审批走终端自己进程的 `canUseTool`、不经后端,同属盲区**(勿把"审批维度"整体误读为可靠);②"等我输入"对 web 未驱动的会话与 AD-3 同因不可见。即:对经后端的操作可靠、对纯终端进程内存里的等待不可见——**如实告知,不假装全知**;缩小差距同 AD-3(依赖上游 attach)。

**备选评估:**

| 方案 | 覆盖 | 瘦中转 | 裁决 |
| --- | --- | --- | --- |
| A 纯 approval_request 聚合 | 窄(漏"等输入";仅经后端审批) | ✔ 零轮询 | 部分 |
| B **投影读模型(审批表 + StatusDeriver + AD-9 触发点)** | 经后端驱动/拦截的会话的审批 + 输入(纯终端会话为盲区) | ✔ 复用已有信号、零新轮询 | **采纳** |
| C 周期轮询所有会话 transcript | 全(含终端会话) | ✘ 自建重型轮询、违 EP-1 | 否决 |

**与现有关系。** 现有有会话列表(SessionRegistry),**无**"等我"聚合投影——**新增**纯函数投影 + 前端 IA;数据全部复用已有(`approval_request` / StatusDeriver / AD-9),不新建数据源。**超越**现有。

### AD-12 工作目录授权范围与项目文件浏览 [承接 FR-23、FR-07、FR-09]

- **方案。** web 侧可达的工作目录限于操作者显式授权的目录集合(配置化,变更即时生效,收窄时既有范围外目录仅拒新开、不中断已开会话);会话发起、项目文件只读浏览(FR-07)、附件输入(FR-09)统一经此范围校验,范围外 fail-closed 拒绝。控制通道处理器补齐**项目文件只读浏览**能力(授权目录内的文件树与文件内容只读读取)——此前 FR-07 的"浏览项目文件"仅追溯到 transcript 工具消息读取,只覆盖"会话中出现过的内容",不构成真实承载,本版补齐。
- **论证。** 最小暴露(PRD §7)与爆炸半径控制(§11.7)的推论:凭证失窃时损害面被限定在已授权目录。浏览为只读、不提供任意写/删(写操作一律经会话内工具调用走审批链,守 AD-7)。
- **与现有关系。** 范围白名单:**采纳**(现有以目录白名单实现,独立判断其正确);文件浏览承载:**补齐**(设计层此前未闭合)。

## 四、数据模型(高层)

后端轻量存储(非消息库,只存 CLI 没有的最小状态):当前会话指针、设备信任(状态、凭证引用、时间)、审批请求(绑定任务、操作、参数、风险、**创建时刻(悬置起点)**、**有效期**、状态)、推送订阅、**最小审计记录**(web 特有安全动作的元数据)。**会话标题从 transcript 派生、权限档从 transcript 读取、effort 不自存**(瘦中转);消息不落后端库,读 transcript。会话运行时另记录**进入等待状态的时刻(`awaitingSince`)**供悬置时长派生(承接 FR-22),属运行时易失态、可重建、不持久化(EP-1)。**留存治理(NFR-16)**:审计 / 审批等记录设上限与轮转、可清理,不无声无限增长。均最小权限原子写。

## 六、关键流程时序(高层)

- **断线补齐**:客户端携最后序号重连 → 补发窗口内增量;超窗口或跨重启 → 完整重载。
- **审批**:请求授权 → 事件 + 推送 → 操作者决策(校验**有效期**;过期语义暂按 fail-closed 默认拒绝,终定随 PRD OQ-05)→ 执行由主机确认 → 广播 + 审计。
- **实时镜像**:v1 为冷读追平(仅已落定内容);实时(含进行中态)依赖上游 attach 能力(AD-3),不靠自建。
- **设备配对/吊销**:新设备待审 → 授权 → 签发凭证;吊销 → 即时断连 + 拒绝。

## 七、部署架构 [承接 v3]

单一常驻后端进程(独立于连接);边缘接入层负责 TLS 与边缘鉴权(部署者自备);轻量存储随主机。部署形态:个人自托管、团队每人各自单实例(同一部署单元)。启动自检 + 排障文档支撑 30 分钟部署(FR-17、NFR)。

## 十、对现有实现的采纳 / 改进 / 超越(透明对照)

本稿对参考基线的态度一览——采纳因独立判断其良构,非因其为现状:

**采纳(独立判断为良构):** 事件信封 + 序号 + 跨重启标识 + 心跳旁路、契约漂移守卫、集中桥接映射、只读并发保护机制、transcript 作会话消息事实源、分层鉴权 + Host 二选一 + 凭证隔离、设备信任生命周期、Web Push + 备用通道、状态派生方式、工作目录范围白名单(AD-12;其中 FR-07 文件浏览承载为设计层补齐)。

**改进(现有有缺陷或不足):** 并发只读的跨设备 over-broadcast → per-会话定向(AD-5);不支持数据删除 → 两级删除边界(AD-1,Could/纯主权);标题自存 → 从 transcript 派生、effort 不自存(比现有更瘦,AD-1)。

**超越(v3 新需求,现有未做):** 防暴破限速(AD-7,现有明确缺口)、强设备认证 WebAuthn(AD-7)、**审批完整性绑定(AD-7,防中间层篡改所批操作)**、**会话发现"等我"聚合(AD-11,读模型投影,现有仅有会话列表、无紧迫度聚合)**。实时镜像**不再列为超越**——co-driven 因现有技术不可行已否决,主路径依赖上游 attach(暂同现有冷读下界),不靠 web 自建。
