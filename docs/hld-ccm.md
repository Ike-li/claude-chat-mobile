## 一、引言

### 1.1 目的与范围

本文档承接 `PRD-CCM-002` v3.2(提交评审稿,尚待评审通过),面向技术评审与详细设计。覆盖组件划分、关键架构决策(含论证与备选)、数据模型、关键流程、部署;不下沉到类级详细设计,不含界面视觉。

### 1.2 设计原则(承接 v3)

终端等价(下界+上界)、透明性、单操作者/机主即 root、断线重连是主路径、fail-closed、上游耦合收敛、**瘦中转(CC CLI 为唯一数据源)**。两条优先级约束:① **终端等价优先**——"通用最佳实践"与终端等价冲突时守终端等价(安全维度例外:限速/强认证不违反等价);② **瘦中转优先**——CC CLI 本身有的一律从 CLI 取、不自建平行数据源;自建仅限 CLI 没有且 web 必需者,并最小化。每多一个自建数据源即多一处漂移/同步/维护成本。

### 1.3 决策方法

每个架构决策(AD-n)给出:**需求依据**(v3 的 FR/NFR)→ **方案** → **论证与备选** → **与现有参考的关系**(采纳 / 改进 / 超越)。

## 二、架构概览(修正版)

> v1 的架构图有根本错误:它假设终端会话经过后端。真实架构中,**终端的 claude 是用户直接运行的独立进程,默认不经过后端**。这决定了同步与实时镜像的形态。

```text
┌── 移动客户端 ──┐        ┌────────────── 主机(操作者自有)──────────────┐
│  Web / PWA     │◀─TLS──▶│ 边缘接入层 ─▶ 后端服务 ─(Agent SDK)─▶ claude │
│  (浏览器)       │◀Push─  │   (反代/隧道)     │        (web 侧会话)         │
└────────────────┘        │                   └─▶ 元数据/审计/设备(轻量存储)│
                          │                                                 │
                          │  会话消息事实源:claude 的会话记录(transcript) │
                          │  ┌─────────────────────────────────────────┐   │
                          │  │ 终端 claude(用户直接运行,独立进程)      │   │
                          │  │   —— 默认不经后端;web 侧只冷读其记录     │   │
                          │  └─────────────────────────────────────────┘   │
                          └─────────────────────────────────────────────────┘
```

后端七组件:CLI 桥接层、会话管理器、事件流引擎、控制通道处理器、鉴权与设备信任、通知服务、可观测性(职责见第五章)。

## 三、关键架构决策

### AD-1 会话消息事实源与存储 [承接 FR-11/19/20、NFR-16]

- **方案。** 会话消息的事实源采用 **claude 自己的会话记录(transcript)**;后端**不另建消息库**。后端另维护**轻量存储**(会话元数据指针、审计事件、设备信任、推送订阅),用于审计查询(FR-19)与恢复。
- **论证。** transcript 为事实源使 web 与终端**天然共享同一会话、双向可续、不分叉**(直接满足 FR-11);另建消息库会制造第二事实源(v1 SQLite 方案的错误)。
- **瘦中转的落实。** 能从 CLI transcript 派生的一律不自存:**会话标题从首条消息派生**、**权限档从 transcript 记录读取**;CLI 未落盘者(如 **effort 档**)**不自存、不恢复**(接受 CLI 默认档——零自建换零复杂度);后端仅保留 CLI 根本没有的最小状态(当前指针、设备信任、推送订阅、最小审计)。审计**不拔高**为独立查询系统,仅记 web 特有安全动作的元数据。
- **删除边界(FR-20 落地,Could/纯数据主权)。** 定义两级:①删除**本产品可见的会话引用**(默认);②可选删除**底层 transcript 文件**(显式二次确认)。满足"删除暴露到服务上的数据"这一主权诉求(非终端等价——见 v3 §12.2),诚实处理"事实源属 claude"的边界(对应 v3 OQ-07)。
- **读取契约(官方文档校准)。** transcript 的**文件路径**是官方稳定契约(hook 输入 `transcript_path`),但 **JSONL 内部格式(字段/结构)未文档化、无稳定承诺**;故消息读取**走官方公开的 session-messages 读取 API**(具体签名以官方 SDK 文档为准、实现时核定),**不自行解析 JSONL 格式**——只把"路径"当契约,规避格式随版本漂移。这是独立查证官方文档(而非"现有能跑")得出的校准。
- **与现有关系。** **采纳**"transcript 事实源 + 只存元数据"(独立判断其为双向共享的最简正解,也是瘦中转的典范);仅在**数据删除**上**改进**(现有不做);审计**保持最小**、标题**改为派生**、effort **不自存**(比现有更瘦);消息读取走官方 API(**改进**:不绑定 JSONL 格式)。

### AD-2 会话共享与接续 [承接 FR-11]

- **方案。** 会话归属以"其记录存在于对应工作目录"为准,故终端建的会话与 web 建的会话彼此可见、可续。web 接续通过驱动一个 `resume` 会话读取既有记录。
- **与现有关系。** **采纳**(现有以此实现双向互见,经测试验证为正确良构)。

### AD-3 实时镜像活跃会话 [承接 FR-13 —— 理想目标]

**需求依据。** FR-13(理想目标):web 观察活跃会话应看到与终端等价的实时进展(含进行中 thinking/子任务)。**约束**:终端等价优先 + **瘦中转**——不为此自建重型机制。

**问题根因。** 实时进展存在于**驱动 claude 的那个进程的内存**,进行中态不落盘;web 冷读磁盘只能看到已落定内容。障碍是"驱动 SDK 的进程不是后端"——而这**本质是 CC CLI / SDK 是否提供旁观活进程能力的问题,应由上游解决,不该 web 自建数据源去补**。

**备选方案评估:**

| 方案 | 实时性 | 终端等价 | 自建复杂度 | 裁决 |
| --- | --- | --- | --- | --- |
| A 后端成为唯一驱动者 | ✔ 全实时 | ✘ 违反(不再用原生 claude) | 高 | 否决 |
| B **依赖上游 attach 能力** | ✔ | ✔ | **零(从 CLI 拿)** | **首选方向(等上游)** |
| C 协同驱动(co-driven) | ✔ 该会话实时 | ◑ | 高 | **否决(现有技术不可行:需上游未提供的 attach;且偏离原生 claude)** |
| D 冷读磁盘 | 仅已落定(非实时) | ✔ | 低 | **v1 基线** |

**决策:**
- **v1 基线 = D**:纯原生会话 web 只读呈现已落定内容,诚实告知"活跃会话进行中态不可见"。
- **首选缩小差距 = B(依赖上游)**:实时态本应由 CC CLI / SDK 提供(attach / 旁观活进程)。**优先等待并采用上游能力,而非 web 自建**;登记为上游依赖并持续追踪(SP-06)。
- **C(co-driven)已否决**:自建终端侧驱动器 + 双路渲染需 attach 活进程能力(上游未提供)——**现有技术下不可行**,且偏离"用户用原生 claude"、维护复杂度高。它因**技术不可行**被排除,非因价值排序被牺牲,故不构成对"终端等价优先"(§1.2)的违反。
- **不接受现状为需求上限**:FR-13 作为理想目标保留;缩小差距的正当路径是**推动/等待上游**,不是 web 自建。

**与现有关系。** **暂同现有(接受冷读下界)+ 依赖上游演进**。较早稿曾评估自建 co-driven,因现有技术不可行且偏离原生 claude 已否决;实时性依赖上游 attach 能力(SP-06),本期不承诺。

### AD-4 事件流与断线补齐 [承接 FR-03、NFR-10/14]

- **方案。** 统一事件信封承载 server→client 全部事件;每会话单调序号 + 每实例跨重启唯一标识(供客户端重置去重基线);有限缓冲供断线增量补齐,超窗口回退完整重载;高频心跳走非补齐旁路避免挤占缓冲;事件类型集受可执行契约守卫防漂移(real ⊇ mock)。
- **与现有关系。** **采纳**(独立判断此为良构;v1 独立稿亦殊途同归推出信封+序号骨架,现有补足了跨重启标识与心跳旁路两个我曾遗漏的要点)。

### AD-5 多入口并发保护 [承接 FR-12]

- **方案。** 保证单一有效写入者;检测到某入口不能安全写入时,将其**置为只读并告知用户**,防两进程写同一记录致分叉;提供清晰的**接管路径**缓解体验损耗;经静默判据自动解除只读。
- **改进点(对现有已知缺陷)。** 现有的只读状态为**全局广播 + 单一活跃视图**,导致"两设备看不同会话时一方解锁误解另一方"(现有有意不修)。本稿**改进**为 **per-会话 / per-连接的只读状态 + 定向下发**,消除跨设备误解——这是现有自身承认"完整修复需要的方向"。
- **与现有关系。** **采纳**只读保护机制(经测试验证有效)+ **改进**其跨设备缺陷。

### AD-6 核心引擎:CLI 驱动 + 桥接映射层 [承接 NFR-14]

- **驱动(输入侧)。** 用 **Agent SDK 的 `query()`** 驱动**本机 claude**(`pathToClaudeCodeExecutable` 指向本机可执行文件,而非 SDK 捆绑副本);采用**流式输入**(streaming input——持续 yield 用户消息,以支持中断与 `canUseTool`)+ **流式输出**(`includePartialMessages`)。**关键**:后端**不自己 `spawn` 子进程、也不直接操作 stream-json 协议**——启动 claude 子进程与 stream-json 传输均由 **SDK 代劳**,后端只调 `query()`。这是瘦中转在引擎层的体现:连进程启动与协议解析都不自建。
- **映射(输出侧)。** SDK 吐出的消息经**唯一一处映射层**翻译为契约事件——唯一的上游耦合点;处理子任务流过滤、批量增量、模式切换兜底、提问特判等 SDK 真实行为;上游破坏性变更仅此处适配,并经契约回归守护。
- **与现有关系。** **采纳**(独立判断"用 SDK 驱动 + 集中映射"是收敛上游耦合的正解;v1 独立稿亦推出映射层结论)。

> **一句话核心引擎**:用 Agent SDK 的 `query()` 驱动本机 claude(流式输入 + 流式输出);`spawn` 子进程与 stream-json 协议由 SDK 代劳,后端只负责"喂输入消息 + 把 SDK 消息映射成前端事件"。—— 常见的"spawn 本机 claude + stream-json"说法漏了 SDK 这一层:那两件事是 SDK 干的,后端不直接做。

### AD-7 分层鉴权 + 防暴破 + 强认证 + 审批完整性 [承接 NFR-01/02/03/04/17 —— 含新增]

- **分层鉴权(采纳)。** 边缘接入层(反代/隧道/身份代理,fail-closed)+ 应用层,两层独立、按 Host 二选一而非叠加;凭证只从启动 shell 注入、不下发移动端、配置文件最小权限原子写。**采纳**现有的分层与凭证隔离(独立判断其良构)。
- **防暴破限速(新增,NFR-03)。** 鉴权端点须有失败限速 / 退避 / 锁定 / 告警。现有**无**此机制(判为缺口)。架构:对失败尝试按来源计数、指数退避、阈值锁定并计入审计与告警。**超越**现有。
- **强设备认证(新增,NFR-04)。** 演进为设备绑定令牌 + **WebAuthn/passkey** 优先,替代"可转发、撤销粗粒度的静态共享口令为唯一手段";保留静态令牌作为降级/首配路径。**超越**现有。
- **审批完整性(新增,NFR-17)。** 移动端展示待批的操作,须与主机实际执行的操作**绑定一致**——审批请求承载操作指纹(命令+参数+cwd 的完整摘要),执行前校验"所批即所行",展示与执行不一致则 fail-closed 拒绝。防中间层在展示与执行之间篡改。现有**无**此绑定校验(判为缺口)。**超越**现有。

### AD-8 设备信任 [承接 FR-18]

- **方案。** 首次接入显式授权的信任模型;多路审批(终端/命令行/已信任设备远程)、信任变更即时生效与自动解锁、待审队列上限防滥用、审批权恒属已信任设备。
- **与现有关系。** **采纳**(经测试验证的良构);与 AD-7 强认证叠加增强。

### AD-9 通知 [承接 FR-14/15]

- **方案。** 后台推送(标准 Web Push + 自签密钥)+ **备用推送通道**绕过特定平台后台限制;触发时机:仅需人工决策或任务终结时推送(中间过程不扰),含深链;不支持后台推送的平台以重连后状态兜底。
- **与现有关系。** **采纳**(含备用通道的现实补充,独立判断其必要)。

### AD-10 任务状态派生 [承接 v3 第八章状态模型]

- **方案。** 后端由在途轮次、挂起审批/提问、错误/完成信号,**派生**出 v3 定义的**用户视角状态**:运行中/等待审批/等待输入/已完成/已失败/已中止/主机离线。规则:"完成"必为真实终态;状态不可确认时呈现"主机离线"而非假定完成。
- **与现有关系。** **采纳**派生方式(实现良构)+ **确保覆盖** v3 要求的完整状态集(现有派生 5 态,须补齐"主机离线/已中止"等用户须区分的状态)。

## 四、数据模型(高层)

后端轻量存储(非消息库,只存 CLI 没有的最小状态):当前会话指针、设备信任(状态、凭证引用、时间)、审批请求(绑定任务、操作、参数、风险、**有效期**、状态)、推送订阅、**最小审计记录**(web 特有安全动作的元数据)。**会话标题从 transcript 派生、权限档从 transcript 读取、effort 不自存**(瘦中转);消息不落后端库,读 transcript。均最小权限原子写。

---

## 五、组件设计(职责)

CLI 桥接层(AD-6)、会话管理器(生命周期独立于连接、状态派生 AD-10)、事件流引擎(AD-4)、控制通道处理器(输入/审批/中止/设备/历史,每操作广播+审计)、鉴权与设备信任(AD-7/8)、通知服务(AD-9)、可观测性(健康/指标/区分五类状态)。

---

## 六、关键流程时序(高层)

- **断线补齐**:客户端携最后序号重连 → 补发窗口内增量;超窗口或跨重启 → 完整重载。
- **审批**:请求授权 → 事件 + 推送 → 操作者决策(校验**有效期**,过期默认拒绝)→ 执行由主机确认 → 广播 + 审计。
- **实时镜像**:v1 为冷读追平(仅已落定内容);实时(含进行中态)依赖上游 attach 能力(AD-3),不靠自建。
- **设备配对/吊销**:新设备待审 → 授权 → 签发凭证;吊销 → 即时断连 + 拒绝。

---

## 七、部署架构 [承接 v3]

单一常驻后端进程(独立于连接);边缘接入层负责 TLS 与边缘鉴权(部署者自备);轻量存储随主机。部署形态:个人自托管、团队每人各自单实例(同一部署单元)。启动自检 + 排障文档支撑 30 分钟部署(FR-17、NFR)。

## 十、对现有实现的采纳 / 改进 / 超越(透明对照)

本稿对参考基线的态度一览——采纳因独立判断其良构,非因其为现状:

**采纳(独立判断为良构):** 事件信封 + 序号 + 跨重启标识 + 心跳旁路、契约漂移守卫、集中桥接映射、只读并发保护机制、transcript 作会话消息事实源、分层鉴权 + Host 二选一 + 凭证隔离、设备信任生命周期、Web Push + 备用通道、状态派生方式。

**改进(现有有缺陷或不足):** 并发只读的跨设备 over-broadcast → per-会话定向(AD-5);不支持数据删除 → 两级删除边界(AD-1,Could/纯主权);标题自存 → 从 transcript 派生、effort 不自存(比现有更瘦,AD-1)。

**超越(v3 新需求,现有未做):** 防暴破限速(AD-7,现有明确缺口)、强设备认证 WebAuthn(AD-7)、**审批完整性绑定(AD-7,防中间层篡改所批操作)**。实时镜像**不再列为超越**——co-driven 因现有技术不可行已否决,主路径依赖上游 attach(暂同现有冷读下界),不靠 web 自建。
