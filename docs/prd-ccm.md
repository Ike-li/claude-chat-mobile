## 一、执行摘要

Claude Code(以下 CC CLI)是运行于开发者本地的自主编程智能体,其价值被"必须在电脑前操作"所限制。本产品建设一套可自托管的移动端 Web 服务,使操作者离开电脑后仍能查看、审批、干预、接续其本机 CC CLI 会话。

**核心原则:终端等价性**,既是下界(不残缺)也是上界(不膨胀)——凡在终端(交互式 `claude` 的 TUI 界面,精确定义见术语表)能得到的,移动端须给出语义等价物;凡它没有的,移动端不新增。`[等价]`

在此原则下,移动端还须解决终端本身不存在、因"把会话搬到公网手机"而新生的问题:防暴力破解与强设备认证(安全基线)、断线重连作为主路径、审批的长时间悬置(注意力不对称的直接后果)、以及在众多会话中快速定位"哪个在等我"。这些是本稿相对现有实现的主要增量。

---

## 二、背景与问题定义 `[价值]`

CC CLI 运行于开发者本地机器(个人自用的家用电脑,或团队中每人各自的机器),在执行高风险操作前请求人工授权,是持续运行、逐步推进的智能体。约束"必须在电脑前"产生三类损耗:任务因等待授权停滞、离席即失去掌控、有效工作时段被物理绑定;个人开发者尤甚——无人接手,离席即停滞。

移动端要解决四个终端没有的问题,它们是本稿多条需求的推导前提:

- **公网暴露**:会话被推到公网可达的服务上,产生终端场景没有的攻击面与数据暴露面。
- **会话断裂**:移动端锁屏/切后台/断网频繁,断线重连是主路径而非异常分支。
- **小屏弱网**:信息呈现与交互须在小屏、低带宽、单手、低注意力下不失真。
- **注意力不对称**:终端有人守着屏幕,移动端没有——无超时 + 无推送 = 无限枯等;反之,推送若不节制又会把操作者淹没。

## 三、目标与价值

| 目标 | 指标(可观测) | 目标值 | 正当性 |
| --- | --- | --- | --- |
| 会话连续性 | 断网/重复投递/乱序 故障注入场景的通过率(不丢/不重/不误判完成;超出补齐能力时可完整重载) | 100% | `[可靠]` |
| 异步触达 | 需人工决策/完成时的通知送达率 | ≥99%(标准推送平台上限内) | `[价值]` |
| 部署可用性 | 初次自托管者跑通首个会话的中位耗时 | ≤30 分钟 | `[价值]` |
| 安全底线 | 未授权者获得代码执行能力 | 0 | `[安全]` |

**实时感知**不列入上表:它是一个**需求状态**(理想=与终端等价含进行中态;当前下界=已落定内容只读)而非可量化指标,完整表述见 §9.3 与 FR-13a/13b。

**核心价值的未量化空位(诚实登记)。** 本产品的核心价值是**解除空间绑定、减少任务因等待授权而停滞**。它最直接的度量是**授权响应闭环时延**——从"智能体请求授权"到"操作者批复"的人机闭环耗时,移动端应显著优于"必须回到电脑才能批"。注意这不是 NFR-12 的系统投递时延(那只是管道)。但在单操作者、自托管、无遥测、n=1 隐私边界下,该闭环时延**难以自动量化**。故本稿不硬塞拟造数字(前稿"能力对等≤10%"式的臆造已删),而将其显性登记为**未量化的价值假设**,以定性验收替代:操作者能在离席状态下完成本来必须回到电脑才能做的授权/干预决策。是否及如何为其建立轻量度量,见 OQ。

价值:解除空间绑定(个人碎片时间、团队工位外延续)、加快响应、成本可控(自托管、复用既有订阅)。

## 四、产品原则(需求的推导公理)

**4.1 终端等价性(下界 + 上界)** `[等价]`
判据:"此刻在项目目录下用交互式 `claude`(那个全屏 TUI 界面,详见术语表『终端』)会得到什么?"移动端须给出语义等价物。下界=不残缺(该 TUI 界面能做的都要能做,不以削能力换表面安全);上界=不膨胀(它没有的不加)。**等价对象是终端 TUI 呈现的能力与信息(语义),不是主机 shell/桌面环境,也不是 TUI 的终端形式(快捷键/ASCII/斜杠语法)**——"用主机 shell 去 grep transcript / cp 导出 / rm 删会话文件"属桌面环境能力、不构成等价对象;而终端的快捷键/ASCII 属客户端形式,移动端用自身范式实现同语义即可。此界定直接决定第六章对搜索/导出的排除与第十二章删除改挂纯主权。等价含**时间维度**——"智能体需要我"到"我知道"的时延,推出推送通知的必要。

**4.2 透明性** `[等价]`
中间层不编辑、不总结、不"优化"智能体输出;上游返回什么就显示什么(含错误原文);上下文管理是智能体自己的职责。失败态如实呈现是其直接推论。

**4.3 单操作者、机主即 root** `[安全]`
每部署实例服务单一操作者;已鉴权请求拥有"操作者坐在终端前"的全部权力——这是产品目的而非漏洞。安全的全部重心因此在**鉴权边界**,而非限制已鉴权者能做什么。

**4.4 断线重连是主路径** `[可靠]`
移动端锁屏/切后台/断网频繁,重连须是常态化的顺畅路径,而非异常处理。

## 五、用户与场景 `[价值]`

**用户(按对需求有区分力的轴刻画,而非按身份堆叠)。** 本产品是单操作者自托管,"个人开发者"与"团队里各自部署的开发者"在需求上没有区别(都是 n=1 实例、彼此独立),故不以此分类——那只是看起来专业的堆叠。真正产出不同需求的是两条轴:

- **熟练度**:*初次自托管者* 依赖安全默认值与启动自检把他安全带过第一次部署,这条轴导出了 FR-18/FR-19 与"≤30 分钟部署"目标;*老手* 追求日常接续的顺畅与低摩擦。
- **使用形态**:*重度远程者*(几乎只用手机、很少回到主机)使"删除留在服务上的数据"(FR-20)从便利升为实打实的主权诉求,也最吃"快速定位哪个会话在等我"(FR-21)与推送触达;*在席为主、偶尔离席者* 则更看重接续时状态不失真。

核心场景(按 MVP 权重排序,此为产品判断,非既有事实):

1. **离席远程接续与审批** —— 最高频、最直接命中痛点(注意力不对称 + 授权停滞)。**MVP 核心**。
2. **即时发起任务 + 被动通知触达** —— 碎片时间价值(通勤发起、睡前处理决策)。**MVP 核心**。
3. **完全脱离电脑独立工作**(浏览文件、多文件 diff 审阅、多轮工具调用) —— 价值高但重,可在核心接续/审批闭环稳固后深化。

个人情境示例:下班回家批一个卡住的授权、通勤看进展、周末发起任务、睡前处理决策、人机分离。

## 六、范围与非目标(守终端等价上界)

**纳入:** 三个核心场景完整能力、安全自托管、设备信任、会话数据管理、移动端信息架构与关键交互(含多会话的发现与定位)。

**非目标(基于产品原则,非架构妥协):**
- 多租户/跨用户隔离/组织权限/计费配额 —— `[等价]` 非本产品(单操作者);
- 中间层智能(摘要/自动重试编排/上下文管理)—— `[等价]` 违背透明性(4.2);
- 终端模拟器、远程桌面复刻、替代/重写 CC CLI —— `[等价]` 越界;
- 原生应用(仅 Web/PWA,Service Worker 只为推送);移动端本地运行智能体、无人监管任意执行;多人同会话协作;本期用量/成本可视化。

**守上界的显式判定:**
- **消息搜索 / 导出**:终端 TUI 本身不提供(在主机 shell 里 grep/cp 属桌面能力、非 TUI 能力),依终端等价上界**不纳入**;若日后判为移动端正当增强,须作为对上界的显式例外评审,而非因"底层难做"而回避。
- **会话数据的存储形态不写入需求。** 需求只规定结果(见第九、十二章),存储形态交由架构裁量。

## 七、功能需求(规范性,不含实现)

完整清单见附录 A。概述:

- **会话接续与观察**:观察会话的状态、进展、最近工具操作、错误与待处理项;识别每个进展的来源(终端/移动端);**在众多会话中发现并定位需要操作者的会话**——尤其把"等待我审批/输入"的会话聚合呈现(注意力不对称的直接要求:操作者要能一眼找到"哪个在等我")。
- **独立工作**:浏览项目文件、逐块审阅多文件改动(小屏不失真)、完成多轮工具调用、向任务提供文件/图片。
- **任务控制**:发起任务、追加约束、回答提问、审批/拒绝(语义与终端逐项一致)、**中止**运行中任务。审批可能在移动端长时间悬置(终端不存在的新情形),故须让操作者看到**审批已悬置的时长**,并在悬置期间如实反映主机/会话状态的变化,避免对一个可能已失效的操作误批(时效的具体机制见 OQ-05)。
- **通知与触达**:需人工决策或任务终结时主动推送,含定位信息、可深链;**须节制,避免对同一会话重复/高频打扰**;应用被回收后重开仍见上下文。
- **安全自托管**:默认最小暴露、启动自检、新设备显式授权与吊销、安全审计可查询。
- **数据管理**(见第十二章):删除自己留在服务上的会话数据(数据主权,Could)。

## 八、任务状态模型(用户视角) `[价值]`

用户视角下,移动端**必须能明确区分**任务处于何种状态(注意力不对称问题的直接要求),故需求层规定用户可观察的状态语义:

| 状态 | 用户含义 |
| --- | --- |
| 运行中 | 智能体在推进 |
| 等待我审批 | 停下等待对高风险操作授权 |
| 等待我输入 | 停下等待回答或补充 |
| 已完成 / 已失败 / 已中止 | 真实终态 |
| 主机离线 | 无法确认状态,仅可查看诊断 |

**规则:**"已完成"必须是**真实终态**,绝不得表示"移动端失联";无法确认状态时须显示为**不确定**(主机离线)而非假定完成(`[可靠]` fail-closed 的状态呈现)。"等待我审批/输入"须附带**已悬置时长**,以支撑操作者对长悬置项的判断(见第七章审批时效)。具体状态如何派生/持久化是实现细节,不在本章。

## 九、会话共享、接续与实时镜像

**9.1 双向共享与接续** `[等价]`
web 与终端共享同一会话集:终端发起的会话在 web 可见、可接续,反之亦然。(**如何**共享——共享同一份会话记录,还是后端统一中继——是架构决策,不在需求。)

**9.2 多入口并发保护** `[可靠]`
会话可被多入口(终端、一或多个移动设备)同时观察与操作。系统须保证:**任一时刻只有一个有效写入者;并发不得导致会话数据损坏或分叉;当某入口不能安全写入时,须将其明确置为只读并告知用户**,而非静默丢弃或制造冲突。(如何实现此保护属机制,需求只要求上述结果。)

**9.3 实时镜像活跃会话(下界 + 理想)** `[等价]`
- **当前可达下界(FR-13a,Must)**:web 观察一个活跃会话时,须只读呈现**已落定**的内容与增量进展——这是终端等价必须守住的底线,也是现状可达的。
- **理想目标(FR-13b)**:进一步看到与终端 TUI 所见**等价**的实时进展,包括进行中的思考与子任务。
- **根本张力(诚实登记)**:理想与"操作者在终端使用**原生** CC CLI"存在张力——原生终端进程的实时内存态未必对外可得,进行中的思考/子任务在原生架构下不可见。本稿将此差距作为**待解的现实差距**(第十四章)登记,交由架构探索,而**不接受它为需求上限**。

## 十一、安全需求

**11.1 分层鉴权** `[安全]`
远程访问须经独立的两层——边缘接入层(反代/隧道/身份代理)与应用层——任一失效不致裸奔;两层 fail-closed。

**11.2 防暴力破解** `[安全]`
暴露远程代码执行能力的服务,其鉴权端点**必须**具备防暴破能力(失败限速 / 退避 / 锁定)。**这是安全基线,不因"依赖强令牌 + 二次认证"而豁免。**(现有实现的定位见附录 E。)

**11.3 强设备认证** `[安全]`
远程访问凭证应采用**设备绑定的强认证**(设备绑定令牌 + WebAuthn/passkey 优先),避免以"可转发、无法细粒度撤销的静态共享口令"作为唯一手段。(强认证不违反终端等价,属安全维度。)

**11.4 设备信任生命周期** `[安全]`
新设备首次接入须经显式授权;授权可远程撤销并即时生效;授权与撤销留审计。

**11.5 失败关闭** `[安全]`
无法确认身份、主机或会话状态时,默认拒绝执行,仅允许查看诊断。

**11.6 透明与凭证隔离** `[安全]`
凭证(密钥/令牌/环境变量)不下发移动端、不入普通日志;敏感输出脱敏;审批展示完整操作 + 工作目录 + 参数 + 风险。

**11.7 上线前安全测试** `[安全]`
威胁建模、权限绕过、重放攻击、敏感信息泄漏测试为发布强制门槛。最高权限档(整机爆炸半径)默认关、无自动进入路径。

**11.8 审批完整性(所批即所行)** `[安全]`
移动端展示给操作者审批的操作,**必须与主机实际将执行的操作绑定一致**;中间层不得在"展示"与"执行"之间篡改操作、参数或工作目录。二者不一致时**须失败关闭、拒绝执行**。这是透明性(4.2)在授权链上的安全推论——对一个远程授权高风险代码执行的系统,"你批准的就是将执行的"是核心安全保证。

## 十二、数据需求

**12.1 数据分类** `[安全]`
高敏感(密钥/令牌/环境变量,不进移动端不入日志)、潜在敏感(命令/输出,默认折叠、可全量展开且不删减内容)、安全审计(审批记录,可查询留存)、敏感控制(设备标识,可撤销)。

**12.2 数据主权:删除暴露到服务上的数据** `[主权]`
web 把会话数据推到了一个**公网可达的服务**上——这是终端场景没有的新暴露面(见第二章)。**用户应能删除自己留在该服务上的会话数据**,作为这一新暴露面的对偶(暴露了就得能收回)。此正当性是**数据主权**,不依赖终端等价——按 4.1 口径,"用主机 shell 删 transcript 文件"是桌面能力、非终端 TUI 能力,故删除并非等价需求。鉴于本产品为单操作者、机主即 root(可直接访问主机文件系统),web 删除属**便利性主权口子**,优先级定为 **Could**;但对"重度远程、很少回主机"的画像(见第五章),它是实打实的诉求,评审时可据画像权重上调。(删除**如何**落到底层存储是实现问题;边界见 OQ-06。)

**12.3 留存有界** `[可靠]`
会话相关数据的磁盘占用须可治理(上限/轮转/可清理),不无声无限增长。

## 十三、验收与测试策略 `[可靠]`

测试只验证外部可观察行为。层级:状态语义与转换、指令/审批/中止、断线重连补齐、多入口并发保护(不损坏/正确置只读)、**多会话发现与定位("等我"聚合正确,验收范围=web 经手会话,见 §14.6)**、设备信任与撤销、通知与深链、**审批长悬置的时效呈现**、敏感数据保护、端到端核心链路、**防暴破与实时镜像差距的专项验证**。

发布门槛(P0 = Must):全部 Must 功能通过 E2E;关键安全/可靠性需求无高危缺陷;断网/重复投递/审批悬置过期/设备撤销/**鉴权暴破**故障注入通过。

## 十四、已知张力与现实差距(诚实登记)

登记为**需求与现实的差距 / 待解张力**,不视为需求上限:

1. **实时镜像 vs 原生终端**(§9.3):理想是等价实时;原生架构下进行中态不可见。**待架构探索**(如让会话事件流对 web 可得的机制),非需求让步。
2. **多入口并发的用户体验**:置只读虽保数据安全,但"我明明想接管"的体验损耗须由清晰的接管路径缓解。
3. **审批时效的定义空白**:移动端引入的长悬置审批,其"过期/失效/需重新确认"语义尚无定论(见 OQ-05);在定论前,至少须如实呈现悬置时长与状态变化,不让操作者误批已失效操作。
4. **上游 SDK 约束**:某些能力(如账号级用量额度)受 SDK 是否暴露所限;须以适配面隔离,不让其污染需求。
5. **删除与"共享会话记录"架构的张力**:若架构选择共享终端的会话记录,则"删除"须谨慎定义边界(删的是本产品的视图,还是底层记录)——12.2 落地时的设计问题,不改变"用户有权删除自己数据"的需求。
6. **FR-21 会话发现的覆盖差额**:纯终端会话的审批走终端进程自身授权、不经后端,其"等我"状态是**结构性盲区**;故 FR-21 的 Must 交付**限于 web 经手(后端驱动/拦截过)的会话**,纯终端盲区的缩小与实时镜像同源——依赖上游 attach(见张力 1),本期不承诺。此为需求覆盖面的诚实收敛,非降低 FR-21 的必要性。

## 附录 D · 立场纠偏史与独立性自查

> 本附录集中容纳"本稿如何形成、为何独立"的过程性叙述。把它下沉至此,是为让正文回归纯需求规格——正文陈述"需求是什么 + 为什么成立",过程辩论不反复出现在每一章。

**纠偏史。** 前两稿中,v2 被现有实现"欺骗":把实现机制、架构决策、约束下的妥协当成了需求,并错误地以"现有代码通过测试"为需求正当性的证据——但**测试通过只证明实现自洽,不证明需求正确**。v3 系列纠正之:需求的对错由独立第一性原理判断,与"现有是否这么做"无关。据此,v3.x 依次剥离了渗入需求层的实现术语(seq/epoch/环形缓冲)与架构决策(如"只读 CLI 会话记录、不自建存储"本是架构裁量,被 v2 写成了需求),补齐了被现有妥协掩盖的安全基线(限速、强认证、审批完整性绑定),恢复了被"实现只派生 5 状态"砍掉的用户视角状态模型,并把"实时镜像"从"接受现状为上限"还原为"理想目标 + 诚实登记差距"。删除了无据量化(如臆造的"能力对等≤10%")。

**独立性自查(诚实性声明)。** 本稿功能需求中多数与现有实现高度吻合(见附录 E)。这个高吻合分布本身须警惕:它既可能是"好设计自然收敛"(独立推导与良构实现殊途同归),也可能是"被现有锚定后反向合理化"(拿现有当答案、再补挂正当性)——v2 正是栽在后者。二者从结果上难以区分,故本稿不以"吻合数量"自证独立,而作如下自查:

- **强独立收敛**(换人从零推大概率也得到):终端等价上下界、透明性、fail-closed、断线重连主路径、单一有效写入者——这些从"公网 + 断线 + 远程授权"的问题定义可直接演绎,不依赖看现有。
- **可能受锚定、须评审复核**:审批三档语义的具体划分、事件来源标识的必要性、审计范围的边界——这些的"恰好如此"更可能参照了现有。评审时应对这些逐条追问"不看现有会不会也这么定"。

承认这一点,比罗列"印证"标签更能支撑"独立"的主张。
