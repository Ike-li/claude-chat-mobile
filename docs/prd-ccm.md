## 方法论:需求正当性与现有参考(取代"测试验证"图例)

v2 用"✓测试验证 = 可安全吸收"标注需求——这是逻辑错误:测试通过只证明**实现自洽**,不证明**需求正确**。本稿改为两栏分离:

**需求正当性来源**(左栏,决定这条需求成不成立,靠独立判断):
`[等价]` 终端等价推导 · `[安全]` 安全第一性原理 · `[可靠]` 可靠性第一性原理 · `[价值]` 用户价值 · `[主权]` 数据主权

**现有实现参考**(右栏,仅事实性参考,不构成正当性):
`{印证}` 现有恰好印证本结论 · `{纠正}` 现有欠妥、本稿据独立判断纠正 · `{超前}` 现有未做到或做不到(理想目标或安全缺口)

> 读法:一条需求即便现有未做(如限速),只要其正当性来源成立(`[安全]`),它就是需求;一条现有做了的(如并发保护),本稿只取其**用户可观察的规范诉求**,不取其**实现机制**。

**对高吻合度的自我审视(诚实性声明)。** 附录 A 中 20 条功能需求有 18 条现有"印证"——独立第一性原理推导却与现有实现高度吻合,这个分布本身须警惕:它既可能是"好设计自然收敛"(独立推导与良构实现殊途同归),也可能是"被现有锚定后反向合理化"(拿现有当答案、再补挂正当性)——而 v2 稿正是栽在后者。二者从结果上难以区分,故本稿不以"18 印证"自证独立,而作如下自查:①**强独立收敛**(换人从零推大概率也得到):终端等价上下界、透明性、fail-closed、断线重连主路径、单一有效写入者——这些从"公网 + 断线 + 远程授权"的问题定义可直接演绎,不依赖看现有;②**可能受锚定、须评审复核**:审批三档语义的具体划分、事件来源标识的必要性、审计范围的边界——这些的"恰好如此"更可能是参照了现有。评审时应对第②类逐条追问"不看现有会不会也这么定"。承认这一点,比 18 个"印证"标签更能支撑"独立"的主张。

## 一、执行摘要

Claude Code(以下 CC CLI)是运行于开发者本地的自主编程智能体,其价值被"必须在电脑前操作"所限制。本产品建设一套可自托管的移动端 Web 服务,使操作者离开电脑后仍能查看、审批、干预、接续其本机 CC CLI 会话。

**核心原则:终端等价性**,既是下界(不残缺)也是上界(不膨胀)——凡在终端(交互式 `claude` 的 TUI 界面,精确定义见术语表)能得到的,移动端须给出语义等价物;凡它没有的,移动端不新增。`[等价]{印证}`

本稿相对被现有实现带偏的 v2 的关键校准:① 需求文档不规定实现机制;② 不接受现有的安全妥协(**补齐防暴破限速、强设备认证**为需求);③ **实时镜像活跃会话**作为终端等价的下界理想目标写入(诚实标注其与"终端用原生 `claude`"的张力),而非接受现状为需求上限。

---

## 二、背景与问题定义 `[价值]`

CC CLI 运行于开发者本地机器(个人自用的家用电脑,或团队中每人各自的机器),在执行高风险操作前请求人工授权,是持续运行、逐步推进的智能体。约束"必须在电脑前"产生三类损耗:任务因等待授权停滞、离席即失去掌控、有效工作时段被物理绑定;个人开发者尤甚——无人接手,离席即停滞。

移动端要解决四个终端没有的问题:**公网暴露**、**会话断裂**(断线重连是主路径而非异常分支)、**小屏弱网**、**注意力不对称**(无超时 + 无推送 = 无限枯等)。`{印证}` 这四点是本稿多条需求的推导前提。

## 三、目标与价值

| 目标 | 指标 | 目标值 | 正当性 |
| --- | --- | --- | --- |
| 会话连续性 | 断线重连后事件不丢、不重、不误判完成 | 完全满足;超出断线补齐能力时可完整重载 | `[可靠]` |
| 实时感知 | web 观察活跃会话看到的进展 | 理想:与终端等价(含进行中状态);当前下界:已落定内容 | `[等价]{超前}` |
| 异步触达 | 需人工决策/完成时的通知送达率 | ≥99%(标准推送平台) | `[价值]` |
| 部署可用性 | 新用户跑通首个会话中位耗时 | ≤30 分钟 | `[价值]` |
| 安全底线 | 未授权者获得代码执行能力 | 0 | `[安全]` |

价值:解除空间绑定(个人碎片时间、团队工位外延续)、加快响应、成本可控(自托管、复用既有订阅)。

## 四、产品原则(需求的推导公理)

**4.1 终端等价性(下界 + 上界)** `[等价]{印证}`
判据:"此刻在项目目录下用交互式 `claude`(那个全屏 TUI 界面,详见术语表『终端』)会得到什么?"移动端须给出语义等价物。下界=不残缺(该 TUI 界面能做的都要能做,不以削能力换表面安全);上界=不膨胀(它没有的不加)。**等价对象是终端 TUI 呈现的能力与信息(语义),不是主机 shell/桌面环境,也不是 TUI 的终端形式(快捷键/ASCII/斜杠语法)**——"用主机 shell 去 grep transcript / cp 导出 / rm 删会话文件"属桌面环境能力、不构成等价对象;而终端的快捷键/ASCII 属客户端形式,移动端用自身范式实现同语义即可。此界定直接决定第六章对搜索/导出的排除与第十二章删除改挂纯主权。等价含**时间维度**——"智能体需要我"到"我知道"的时延,推出推送通知的必要。

**4.2 透明性** `[等价]{印证}`
中间层不编辑、不总结、不"优化"智能体输出;上游返回什么就显示什么(含错误原文);上下文管理是智能体自己的职责。失败态如实呈现是其直接推论。

**4.3 单操作者、机主即 root** `[安全]{印证}`
每部署实例服务单一操作者;已鉴权请求拥有"操作者坐在终端前"的全部权力——这是产品目的而非漏洞。安全的全部重心因此在**鉴权边界**,而非限制已鉴权者能做什么。

**4.4 断线重连是主路径** `[可靠]{印证}`
移动端锁屏/切后台/断网频繁,重连须是常态化的顺畅路径,而非异常处理。

## 五、用户与场景 `[价值]`

用户:个人开发者(自用,下班/通勤/居家远程照看)、团队中的开发者(每人各自一套单用户实例,彼此独立)、初次自托管用户(依赖安全默认值)。

核心场景:离开电脑后远程接续、完全脱离电脑独立工作、即时发起任务 + 被动通知。个人情境:下班回家批一个卡住的授权、通勤看进展、周末发起任务、睡前处理决策、人机分离。

## 六、范围与非目标(守终端等价上界)

**纳入:** 三个核心场景完整能力、安全自托管、设备信任、会话数据管理、移动端信息架构与关键交互。

**非目标——基于产品原则,非架构妥协:**
- 多租户/跨用户隔离/组织权限/计费配额 —— `[等价]` 非本产品(单操作者);
- 中间层智能(摘要/自动重试编排/上下文管理)—— `[等价]` 违背透明性(4.2);
- 终端模拟器、远程桌面复刻、替代/重写 CC CLI —— `[等价]` 越界;
- 原生应用(仅 Web/PWA,Service Worker 只为推送);移动端本地运行智能体、无人监管任意执行;多人同会话协作;本期用量/成本可视化。

**守上界的显式判定(区别于 v2 被架构妥协带偏的"非目标"):**
- **消息搜索 / 导出**:终端 TUI 本身不提供(在主机 shell 里 grep/cp 属桌面能力、非 TUI 能力),依终端等价上界**不纳入**;若日后判为移动端正当增强,须作为对上界的显式例外评审——而非因"底层难做"而回避。
- **会话数据的存储形态不写入需求。** v2 曾把"只读 CLI 会话记录、不自建存储"当成需求——那是架构决策。本稿需求只要求结果(见第九、十二章),存储形态交由架构裁量。`{纠正}`

## 七、功能需求(规范性,不含实现)

完整清单见附录 A。概述:

- **会话接续与观察**:移动端观察会话的状态、进展、最近工具操作、错误与待处理项;识别每个进展的来源(终端/移动端)。
- **独立工作**:浏览项目文件、逐块审阅多文件改动(小屏不失真)、完成多轮工具调用、向任务提供文件/图片。
- **任务控制**:发起任务、追加约束、回答提问、审批/拒绝(语义与终端逐项一致)、**中止**运行中任务。
- **通知与触达**:需人工决策或任务终结时主动推送,含定位信息、可深链;应用被回收后重开仍见上下文。
- **安全自托管**:默认最小暴露、启动自检、新设备显式授权与吊销、安全审计可查询。
- **数据管理**(见第十二章):删除自己留在服务上的会话数据(数据主权,Could)。

## 八、任务状态模型(用户视角) `[价值]{纠正}`

> v2 因现有"实现只派生 5 状态"就砍掉了状态定义——那是实现视角。用户视角下,移动端**必须能明确区分**任务处于何种状态(注意力不对称问题的直接要求),故需求层须规定用户可观察的状态语义:

| 状态 | 用户含义 |
| --- | --- |
| 运行中 | 智能体在推进 |
| 等待我审批 | 停下等待对高风险操作授权 |
| 等待我输入 | 停下等待回答或补充 |
| 已完成 / 已失败 / 已中止 | 真实终态 |
| 主机离线 | 无法确认状态,仅可查看诊断 |

**规则:**"已完成"必须是**真实终态**,绝不得表示"移动端失联"; 无法确认状态时须显示为**不确定**(主机离线)而非假定完成(`[可靠]` fail-closed 的状态呈现)。具体状态如何派生/持久化是实现细节,不在本章。

## 九、会话共享、接续与实时镜像

**9.1 双向共享与接续** `[等价]`
web 与终端共享同一会话集:终端发起的会话在 web 可见、可接续,反之亦然。(**如何**共享——共享同一份会话记录,还是后端统一中继——是架构决策,不在需求。v2 把"读 CLI 记录"写进需求是越界。`{纠正}`)

**9.2 多入口并发保护** `[可靠]`
会话可被多入口(终端、一或多个移动设备)同时观察与操作。系统须保证:**任一时刻只有一个有效写入者;并发不得导致会话数据损坏或分叉;当某入口不能安全写入时,须将其明确置为只读并告知用户**,而非静默丢弃或制造冲突。(现有以"只读镜像锁"实现此需求 `{印证}`——但镜像锁是机制,需求只要求上述结果。)

**9.3 实时镜像活跃会话(理想目标 + 张力)** `[等价]{超前}`
- **目标需求**:web 观察一个**活跃**会话时,应看到与在终端 TUI 界面所见**等价**的实时进展,包括进行中的思考与子任务——因为这正是终端等价的下界(TUI 界面本就能实时看到)。
- **根本张力(诚实登记)**:这与"操作者在终端使用**原生** CC CLI"存在张力——原生终端进程的实时内存态未必对外可得。需求陈述此目标不等于承诺现有架构能达到。
- **当前可达下界**:只读呈现**已落定**的内容;进行中的思考/子任务在原生架构下不可见。本稿将差距作为**待解的现实差距**(第十四章)登记,交由架构探索,而**不接受它为需求上限**。

## 十一、安全需求

**11.1 分层鉴权** `[安全]`
远程访问须经独立的两层——边缘接入层(反代/隧道/身份代理)与应用层——任一失效不致裸奔;两层 fail-closed。`{印证}`

**11.2 防暴力破解(硬需求,纠正现有)** `[安全]{纠正}`
暴露远程代码执行能力的服务,其鉴权端点**必须**具备防暴破能力(失败限速 / 退避 / 锁定)。**这是安全基线,不因"依赖强令牌 + 二次认证"而豁免。** 现有实现未提供,本稿判定为**安全缺口须补**——v2 曾错误地将其降格为"分歧点"去尊重现有的"非目标"。

**11.3 强设备认证(纠正现有)** `[安全]{纠正}`
远程访问凭证应采用**设备绑定的强认证**(设备绑定令牌 + WebAuthn/passkey 优先),避免以"可转发、无法细粒度撤销的静态共享口令"作为唯一手段。现有以静态共享令牌为主,本稿判定应演进。(强认证不违反终端等价,属安全维度。)

**11.4 设备信任生命周期** `[安全]`
新设备首次接入须经显式授权;授权可远程撤销并即时生效;授权与撤销留审计。`{印证}`

**11.5 失败关闭** `[安全]`
无法确认身份、主机或会话状态时,默认拒绝执行,仅允许查看诊断。`{印证}`

**11.6 透明与凭证隔离** `[安全]{印证}`
凭证(密钥/令牌/环境变量)不下发移动端、不入普通日志;敏感输出脱敏;审批展示完整操作 + 工作目录 + 参数 + 风险。

**11.7 上线前安全测试** `[安全]`
威胁建模、权限绕过、重放攻击、敏感信息泄漏测试为发布强制门槛。最高权限档(整机爆炸半径)默认关、无自动进入路径。

**11.8 审批完整性(所批即所行)** `[安全]{超前}`
移动端展示给操作者审批的操作,**必须与主机实际将执行的操作绑定一致**;中间层不得在"展示"与"执行"之间篡改操作、参数或工作目录。二者不一致时**须失败关闭、拒绝执行**。这是透明性(4.2)在授权链上的安全推论——对一个远程授权高风险代码执行的系统,"你批准的就是将执行的"是核心安全保证。现有实现未做此绑定校验,本稿判定为**安全缺口须补**。

## 十二、数据需求

**12.1 数据分类** `[安全]{印证}`
高敏感(密钥/令牌/环境变量,不进移动端不入日志)、潜在敏感(命令/输出,默认折叠、可全量展开且不删减内容)、安全审计(审批记录,可查询留存)、敏感控制(设备标识,可撤销)。

**12.2 数据主权:删除暴露到服务上的数据** `[主权]{超前}`
web 把会话数据推到了一个**公网可达的服务**上——这是终端场景没有的新暴露面(见第二章)。**用户应能删除自己留在该服务上的会话数据**,作为这一新暴露面的对偶(暴露了就得能收回)。此正当性是**数据主权**,不依赖终端等价——按 4.1 口径,"用主机 shell 删 transcript 文件"是桌面能力、非终端 TUI 能力,故删除并非等价需求。鉴于本产品为单操作者、机主即 root(可直接访问主机文件系统),web 删除属**便利性主权口子**(对不便直接操作主机的自托管用户有真实价值),优先级定为 **Could**。(删除**如何**落到底层存储是实现问题;边界见 OQ-07。)

**12.3 留存有界** `[可靠]`
会话相关数据的磁盘占用须可治理(上限/轮转/可清理),不无声无限增长。

## 十四、已知张力与现实差距(诚实登记)

区别于 v2 把这些当"可接受的边界",本稿登记为**需求与现实的差距 / 待解张力**,不视为需求上限:

1. **实时镜像 vs 原生终端**(第九章 9.3):目标是等价实时;原生架构下进行中态不可见。**待架构探索**(如让会话事件流对 web 可得的机制),非需求让步。
2. **多入口并发的用户体验**:置只读虽保数据安全,但"我明明想接管"的体验损耗须由清晰的接管路径缓解。
3. **上游 SDK 约束**:某些能力(如账号级用量额度)受 SDK 是否暴露所限;须以适配面隔离,不让其污染需求。
4. **删除与"共享会话记录"架构的张力**:若架构选择共享终端的会话记录,则"删除"须谨慎定义边界(删的是本产品的视图,还是底层记录)——这是 12.2 落地时的设计问题,不改变"用户有权删除自己数据"的需求。

## 十五、实施与风险(择要)

## 附录 B · 术语表(择要)

CC CLI(本文对 Claude Code 的简称;其**交互形态**是一个 TUI——`claude` 命令启动的全屏终端界面)、Claude Agent SDK、终端等价性(下界+上界)、透明性、单操作者、fail-closed、断线重连主路径、实时镜像、多入口并发保护、数据主权、可追溯矩阵、MoSCoW/P0。

**『终端』的精确所指(消歧义):** 本文所称"终端"特指**交互式 `claude` 的 TUI 界面**(呈现消息流、审批、thinking、diff、状态、命令效果);**终端等价 = 与该 TUI 呈现的能力与信息(语义)等价**——不是与 `claude -p`(headless 纯 CLI 调用)等价,**不含** TUI 的终端形式(快捷键/ASCII 渲染/斜杠命令语法),**亦不含**主机 shell/桌面环境。移动端以自身范式(按钮/手势/选择器)实现同语义。

(**注:seq/epoch/镜像锁/事件信封/环形缓冲等为实现术语,属架构文档,不在需求术语表。**)
