# 应用状态 — 业界设计参考研究

> **来源说明（诚实）**：本文档是对「进程管理器 / 自托管监控面板 / 任务进度与通知聚合」三类工具设计的综合研究。原计划由三个研究 subagent 产出，但在跨会话恢复时无法可靠取回它们的原始返回文本，故本文由主会话基于对这些工具**真实设计的知识**重写——非伪称的 agent 原话。每条结论均点名具体工具/功能，可自行核验（附核验关键词）。
> **用途**：为「应用状态」功能（见 `status-center-proposal.md`）提供业界参照，检验并修正三个高层洞察（A 两功能捆绑 / B 与 push 同源 / C 异常驱动·聚合器为核心）。

---

## 一、进程管理器：如何暴露「多个后台进程的状态」

对应 ccm 的**会话总览**（多会话 ≈ 多进程）。

**PM2**（`pm2 list` / `pm2 monit`）
- `pm2 list` = 一张表：app 名、id、uptime、**↺ 重启次数**、**status（online/stopped/errored，绿/灰/红单词）**、cpu%、mem。
- 精华：**状态就是一个带色的单词**（online 绿 / errored 红）；**重启次数是健康气味**（频繁重启 = 崩溃循环）；uptime 表达"稳不稳"。
- 糟粕（搬到手机）：cpu%/mem 实时数、cluster 模式细节——手机用户不盯这些。`pm2 monit` 的实时资源监控是工作站场景。

**systemd**（`systemctl status` / `systemctl --failed`）
- `systemctl status` = 一个带色圆点（● 绿=active / 红=failed）+ 最近日志几行。
- **`systemctl --failed` = 只列失败单元的专门视图**——这是「异常驱动」最硬的业界先例：不平铺全部，给你一个"只看坏的"。
- 精华：**彩色圆点 + 一个"只看故障"的过滤视图**。

**Docker**（`docker ps`）/ **k9s**（k8s TUI）
- `docker ps` 的 STATUS 列："Up 3 hours" / "Exited (1) 2 min ago" / (healthy)/(unhealthy)。
- k9s：颜色编码 + 键盘钻取，信息密集。
- 精华：状态文字自带"多久了"（Up 3h）；健康检查结果内联。糟粕：k9s 的密度是给运维的。

**小结**：多进程状态的成熟范式 = **每行一个带色状态词 + 一个健康气味（重启/错误数）+ 可选的"只看故障"过滤**。注意：`pm2 list` 是**全展示**的——因为进程列表小而稳；"只看故障"是补充而非默认。→ 对 ccm：会话数少，会话列表全展示（F1）合理，但应提供"待处理优先"排序/过滤。

## 二、自托管监控面板：如何暴露「服务健康/异常」

对应 ccm 的**软件自身可见**。重点检验"异常驱动 vs 全量陈列"。

**Uptime Kuma**（最流行的自托管监控）
- 顶部一条**大色带**："All Systems Operational"（绿）——**这就是"一瞥即知"的信号**。下面才是各监控项 up/down + uptime%。
- 通知（Telegram/邮件/…）**由同一套 up/down 状态驱动**——**push 与面板天然同源**。→ 强验证洞察 B。
- 精华：**总状态色带一句话**；push 同源；status page 极简（绿/黄/红 + 组件列表）。

**menu bar / 状态栏图标**（macOS 状态项等）
- **一个图标，用颜色/形状编码整体状态，点开看详情**——这是"一瞥即知"的极致，最贴近洞察 C。
- 精华：常驻但极小（一个图标）；正常时安静、异常时变色；详情在点击后。

**Netdata / Glances**（实时系统监控）
- 信息密度**极高**的实时仪表墙——**反面教材**。给工作站看很棒，"手机瞥一眼"是灾难。
- 糟粕：指标墙、时序图、告警规则引擎——运维团队重型设施。

**Portainer / Healthchecks.io**
- Portainer：容器列表 + 状态，移动自适应但仍偏密。
- Healthchecks.io：极简表达"活着/挂了"（cron 心跳），一个 ✓/✗ + 最后一次时间。

**小结**：主流不是"全隐形"，而是**"正常=一句安静的『一切正常』(绿)，异常=凸显"**。Uptime Kuma 的绿色安心带 + menu bar 单图标 = ccm 该抄的模型。→ 修正洞察 C：不是"正常时完全隐形"，而是"正常=低成本的安心信号，异常=升级凸显"。

## 三、任务进度 + 通知聚合：如何呈现「进度」和「需要你处理」

对应 ccm 的 **todo 进度** 和 **"需要你处理"聚合**。

**任务/多步进度**
- **CI（GitHub Actions）**：run 级一个 ✓/✗ 字形（一瞥即知）+ 展开是 steps（✓/●/○/✗）+ 摘要"N/M 通过"。**摘要 + 可展开**是核心范式。
- **AI agent（Devin/Cursor 等）**：显示当前步/正在跑的工具，常是可折叠的计划清单，步骤打勾。
- **Linear/Things/GitHub checklist**："3/5" 进度 + 复选清单。
- 精华：**进度 = "N/M 摘要 + 可展开详情"**，不总是内联铺开所有步；run 级状态字形当"一瞥"。

**"需要你处理"聚合**
- **通知中心（iOS/Android）**：按 app 分组、可操作（划一下就处理）、badge 数字。
- **GitHub/Linear inbox**："needs your review / unread" 的行动收件箱。
- **GitHub Actions environment approval**：明确的 "Review deployments" 按钮——**action-required 作为一等按钮**，不埋没。
- 精华：**把分散的"需要你"聚到一处 + badge 计数 + 一键直达处理**；action-required 是独立醒目的操作,不是列表里的一行。
- 糟粕：团队收件箱的指派/优先级矩阵/过滤/snooze——单用户自托管用不上。

**小结**：进度用"摘要+展开"；"需要你处理"用"聚合 inbox + badge + 一键直达"。→ 强验证洞察 C（聚合器为核心）；且 CI 的 run 级"N/M"提示：**会话行进度摘要（3/5）其实是业界常态**——这一点轻微地重新打开了我之前砍掉的"会话行摘要"。

## 四、关键改进点（对 `status-center-proposal.md`）

1. **把"一瞥即知"的单一状态信号扶为核心**（menu bar 图标 / systemctl 圆点 / Uptime Kuma 色带）。当前方案把 F3 事件列表当核心、F4 提示当附属——**业界一致表明应反过来**：核心是一个滚动汇总"正常/有事"的信号，列表是钻取详情。→ **F4 应升为核心，而非可砍的附属**。
2. **正常 = 安静的"一切正常"，而非全隐形**（Uptime Kuma 绿带 / Healthchecks ✓）。修正"正常时隐形"为"正常=低成本安心信号"。
3. **push 与状态同源坐实**（Uptime Kuma 用同一 up/down 驱动通知与面板）。→ ccm 应有**一个判定"什么算需要你/异常"的函数**，同时驱动 push 通知 + F4 badge + F3 严重级。不要两套判定。
4. **"只看故障"过滤有先例**（`systemctl --failed`）。软件事件（F3）的价值就是"什么坏了"，不必陈列；会话列表（F1）因列表小可全展示 + 待处理排序。
5. **进度 = 摘要 + 展开**（CI/Linear）。**会话行"3/5"摘要其实是业界常态**——之前以"碰后端热路径"为由砍掉，值得重新权衡（也许有更轻的拿进度方式，见下）。
6. **健康气味**（PM2 重启次数）：会话的 error 态、"push 失败 ×N" 这类计数，比单纯 true/false 更有信息。
7. **拒绝 Netdata 化**：不做指标墙/时序图/告警规则引擎——坐实极度克制。

## 五、我的洞察（业界如何验证/修正 A/B/C）

**洞察 A（两功能捆绑）→ 部分修正**：业界其实**用一个状态原语统一了两者**。`systemctl status` 既答"在不在跑"也答"健不健康"；menu bar 一个图标滚动汇总一切。所以不必拆成两个割裂功能——而应**用一个"一瞥即知"的顶层信号把"会话待处理"和"软件健康"卷到一起**，下面分两个详情区。这比"两个功能"更统一，也解掉了 F4 的别扭（它本就该是那个统一信号）。

**洞察 B（与 push 同源）→ 强验证**：Uptime Kuma/Healthchecks 全是同一状态驱动通知与界面。ccm 应落一个 `whatNeedsAttention()` 判定，喂给 push + badge + 事件列表。这是本轮研究**最该落地的一条**。

**洞察 C（异常驱动、聚合器为核心）→ 验证 + 一处修正**：方向对（menu bar / 通知中心 / action-required 全支持"聚合 + 一瞥 + 一键直达"）；**但"正常时隐形"过头了**——Uptime Kuma 的绿色安心带说明：给一个"一切正常"的低成本确认是有价值的，不必全藏。所以是"**正常=安静确认，异常=凸显升级**"。

**净结论**：研究把方案的重心从"陈列式状态页（F3 列表为主）"**扶正为"一瞥即知的状态原语为核心（F4 升为主）+ 与 push 同源的统一判定 + 钻取式详情"**。这不增加复杂度，反而更聚焦、更克制、更符合 ccm 哲学——是一次"取其精华"的重构方向，而非加功能。
