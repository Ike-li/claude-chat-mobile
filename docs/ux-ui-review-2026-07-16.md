# Claude Chat Mobile — UX / UI 评价

**评价者：** Claude（本会话，模型：grok-4.5[1m] / 环境标注）  
**日期：** 2026-07-16  
**评价对象：** 本仓库当前前端（`public/`，分支 `dev` 工作区状态）  
**输入材料：**
1. 机主提供 6 张真机 UI 截图（首页会话列表、配置面板×2、工作区与会话抽屉、交互日志、带附件空聊）
2. 只读勘察：`public/css/app.css`、`public/index.html`、`public/js/app.js` + `public/js/app/*`、`docs/design.md`、`README.md`

**评价立场：**  
按产品真实定位打分——「已会用 Claude Code CLI 的人，把手机当终端随身入口」；**不是**「第一次用 AI 的大众聊天 App」。  
主锚点 = `docs/design.md` 的**终端语义等价**，不是视觉炫技或通用 SaaS 完成度。

**性质：** 纯评价文档，**不含实现计划、不改代码**。供与其他模型输出横向对比。

---

## 1. 总评

**一句话：**  
UI 已超过多数自托管 / 开源 agent 壳，达到「像产品」；核心工作流 UX（发消息 → 看过程 → 审批 → 多会话 → 断线回来）扎实，且与设计不变量对齐。信息密度与无障碍仍偏「机主专用 IDE」——对 n=1 重度使用已经够好，对开源首次 5 分钟理解成本还有可感门槛。

| 维度 | 分（/10） | 依据摘要 |
|------|----------|----------|
| 视觉识别 / 品牌感 | **8.5** | 暖米白 + 赤陶 accent + Claude 回复 serif，有签名感，不像通用 Bootstrap |
| 移动端基础打磨 | **8.0** | safe-area、16px 防缩放、sheet、44px 伪热区、reduced-motion、键盘 inset |
| 核心任务流 UX | **8.0** | 流式 / 工具卡 / 审批完整性意图 / 断线主路径 /「需要你」与 design.md 对齐 |
| 信息架构（日常 vs 运维） | **6.5** | 聊天面干净；抽屉与配置极密；专家信息正确但默认展开偏重 |
| 一致性 / 细节 polish | **6.5** | 模型卡文案撞车、状态条堆叠、中英混排、附件同名等截图级糙点 |
| 主题 / 国际化产品化 | **5.5** | 系统 dark 有 token；无应用内切换；语法高亮固定 github-light；文案中文硬编码、夹英文 |
| 无障碍 / 键盘 | **4.5** | `data-testid` 多于 `aria-*`；chip/sheet 键盘与 SR 偏弱 |
| 代码可演进性（影响后续 UX 改动成本） | **5.0** | 渲染主战场仍在 ~4.8k 行 `app.js`；`app/*` 模块多是薄封装 |

**综合：7.5 / 10**

| 若目标是… | 水平判断 |
|-----------|----------|
| 机主自己口袋里的 Claude Code | **已经很好**，可日常重度依赖 |
| 开源 README 截图吸人 clone | **足够**；模型卡文案等问题会减分 |
| 零 CLI 经验的大众聊天 | **不对口**（权限档 / 思考强度 / bypass 是特性不是 bug） |
| Claude.ai 级无障碍与全主题 | **差一档** |

**明确不该做的「改进」：**  
为了「更像大众 AI 聊天」削弱权限档、思考强度、工具过程可见性——会违反终端等价。正确方向是：**默认路径更安静，专家信息可展开**。

## 2. 产品语境（打分前提）

| 项 | 结论 |
|----|------|
| 北极星 | 手机打字 ≡ 终端对 claude 打字（形态可不同：气泡代 TUI，sheet 代 y/n） |
| 用户 | 已有 CLI 习惯的 power user；每实例单用户自托管 |
| 非目标 | 多租户聊天、远程桌面镜像、离线 PWA 缓存、xterm 像素复刻 |
| 移动端四问题 | 公网暴露 / 会话碎片化（断线=主路径）/ 小屏弱网 / 注意力非对称 |
| 成功画面 | 一周内夜里想让 Claude 干活不必再走到电脑前 |

因此：配置面板里 6 档权限 + effort + 模型 tile **密集是正确的**；问题在**展示清晰度与默认安静度**，不在「该不该有这些控件」。

## 3. 视觉与设计系统

### 3.1 Token（`public/css/app.css`）

- 画布 `#F4F3EE`、表面 `#FBFAF6`、下沉 `#EFEDE4`
- Accent 赤陶 `#C96442`（及 deep / bright / wash）
- 字色三级 ink / ink-soft / ink-faint
- 语义色：info / success / danger / warning 齐全
- 圆角 lg/md/sm、暖色 shadow（偏棕而非冷灰黑）
- 字体：UI = system sans；**Claude 正文 = Source Serif 4（自托管 woff2）**；mono = SF Mono 栈

### 3.2 签名手笔

| 手法 | 评价 |
|------|------|
| 助手正文 serif + 1.72 行高 | **强**。与用户气泡 sans 分流，阅读感明显区别于 ChatGPT 类壳 |
| 思考块 italic serif 淡色 | 合理，内心独白隐喻清楚 |
| 工具卡左侧 accent 竖条 | Claude 风格过程可见，折叠默认不刷屏 |
| 暖米浅色主调 | 截图观感成熟、克制，不像「工程师深色终端搬网页」 |

### 3.3 主题

- 注释写「浅色单主题」，但存在 `prefers-color-scheme: dark` 的 token 重映射
- **无应用内主题开关**
- 语法高亮固定 `github-light` → 系统 dark 下代码块可能违和
- PWA / manifest 主题色偏浅色奶油

**判断：** 浅色主设计完整且好看；dark 是「系统跟随补丁」而非完整双主题产品。对当前定位可接受，不算硬伤。

## 4. 移动端工程细节

已观察到并认可：

- `viewport-fit=cover`、`env(safe-area-inset-*)`、`.safe-top` / `.safe-bottom`
- 输入 16px 防 iOS 聚焦缩放
- bottom sheet + drag handle；`max-h` 约束
- composer 按钮 `::after` 扩到约 44px 热区
- `touch-action: pan-y`、swipe 关会话
- `prefers-reduced-motion` 关闭入场动画
- 键盘：`interactive-widget=resizes-content` + visualViewport inset 补偿（代码侧有注释应对 iOS 文件选择器坑）
- ≥1024px 侧栏 dock，产品仍明确 phone-first

**判断：** 移动端基础分高，是「真在手机上用」而不是「桌面站缩一下」。

## 5. 分表面评价（结合截图）

### 5.1 首页 / 最近会话（截图 1）

**优点**
- 奶油底 + 圆角卡片 + 顶部项目 pill，第一印象像成品
- 「最近活跃会话」信息够用：标题、工作区、时间、「进入 →」
- 底部 composer 常驻 model / 权限 / effort chip，驾驶控件不藏深

**问题**
- 长标题截断策略一般，扫读成本高
- 整卡可点时右侧「进入 →」略冗余（可保留 chevron，弱化文案 CTA）

**分：7.5**

### 5.2 配置面板（截图 2–3）

**优点**
- 模型 / 权限 / 思考强度用大 tile，触控友好
- 权限六档（默认审批 / 计划 / 自动接受编辑 / 免打扰 / Auto / Bypass）信息架构正确；Bypass 危险语义有区分意图
- 完成提示（提示音 / 震动 / 前台系统通知）+ 试听，对齐「注意力非对称」
- 访问与设备、安全体检、开发者重启分区合理

**问题（截图硬伤）**
- **多张模型卡主标题都是 `grok-4.5`**，仅靠副文案区分 Custom Opus/Sonnet/Haiku → 扫一眼无法选，像配置错误或显示 bug，严重拉低信任感
- 单页超长 scroll，缺锚点 / 分段；开发者区未默认折叠
- 选中态靠描边，当前生效项可以再强一点（勾选 /「使用中」chip）

**分：6.5**（结构 8，展示清晰度被模型卡拖累）

### 5.3 工作区与会话抽屉（截图 4）

**优点**
- 状态图例（运行中 / 待审批 / 出错 / 已完成）教育成本一次付清
- live 实例计数、服务重启横幅 —— **运维可见性做对了**
- 多工作区树 + 新建 + 删除，符合多仓并行 JTBD
- 「需要你」聚合在设计与代码路径上是一等公民（截图未突出展示，属架构优点）

**问题**
- 默认信息过载：图例 + 服务横幅 + live 表 + 树 + 操作同时出现
- 对「只想切会话」路径不友好
- 宜：**需要你 → 会话树** 优先；状态/图例/live 默认折叠

**分：7.0**（能力 8.5，默认层级 5.5）

### 5.4 交互日志（截图 5）

**优点**
- 深色终端感 overlay，与奶油主 UI 分离正确
- 时间戳 + 通道标签（WEB-RECV / CLI-CONN）对排障有用
- 不污染主聊天流

**问题**
- 标题中英双语冗余：「交互日志 (Current Session Trace)」「关闭 (Close)」
- 这是 power-user 工具，完成度够用即可

**分：7.5**

### 5.5 空聊 + 附件 + 状态条（截图 6）

**优点**
- 欢迎文案克制
- 附件 tray 在输入上方，结构清楚
- chip 同步反映权限档 / 思考强度变更

**问题**
- **中央灰字状态条堆叠 3 行**（权限档变化 / 续接会话 / 思考强度）—— 空态应安静，现在像系统日志
- **五张附件全叫 `image.png`**，无法区分
- 「下一条消息生效」类反馈更适合只改 chip，不必再中心刷条

**分：6.0**

### 5.6 聊天主面 / 工具 / 审批（代码 + 产品意图；部分未出现在本批截图）

**优点**
- 用户气泡 vs 助手 serif 分流清晰
- 工具卡 `<details>` 默认折叠 + 状态 emoji/图标
- 审批 sheet 设计意图含完整命令 + cwd（design 硬约束）；完整性预警路径存在
- busy 收进输入 pill，省竖向空间
- 乐观发送、历史与 live 视觉词汇统一

**问题**
- 审批 / question sheet 的 dialog 语义与焦点陷阱弱于附件预览
- 状态 emoji 对读屏不友好
- stream 与 history 双路径存在细微分叉风险（工程债）

**分：8.0**（核心价值面）

## 6. 信息架构总图

```
[鉴权 / TOFU / CF 重登]  ← 安全门，有意摩擦
        ↓
[顶栏] 会话 · 项目 pill · 连接点 · 首页 · 控制台 · 新建
        ↓
[主列]  仪表盘空态 | 会话 transcript（气泡 / thinking / tool / 系统条）
        ↓
[中层条] statusline(可折) · model/perm/effort chips · 活动/任务/镜像 banner
        ↓
[composer] 附件 tray · 输入 pill（含 busy/停止）· 附件/设置/发送
        ↓
[抽屉/Sheet] 工作区树 · 配置 · 审批 · 提问 · 删会话 · 文件浏览 · 日志
```

**判断：** 分层正确。问题不在缺层，在**某些层默认噪音过大**（空态系统条、抽屉 live 全开、配置超长）。

## 7. 无障碍与一致性

| 项 | 现状 | 严重度 |
|----|------|--------|
| chip 可访问性 | 可点 div，缺 role/aria-pressed/键盘 | 中 |
| sheet dialog 语义 | 附件预览有；审批/提问弱 | 中 |
| aria-live | 几乎只有 RTT | 中（流式/状态对 SR 不友好） |
| 中英混排 | 主路径中文 + 夹英文标签 | 低–中（观感） |
| 模型卡标题 | raw id 撞车 | **高（可信度）** |
| 附件命名 | 全 image.png | 中 |
| dark + 代码高亮 | 不匹配 | 低–中 |
| testid vs a11y | E2E 友好、SR 不友好 | 中 |

## 8. 粗糙点优先级清单（供对比：其他模型是否抓到）

### P0 — 截图级、影响信任

1. **模型选择卡主标题撞车**（多卡皆 `grok-4.5`）  
2. **空聊中央状态条堆叠**（像日志墙）  
3. **附件 chip 无法区分**（全 `image.png`）

### P1 — 信息架构

4. 工作区抽屉默认过载（图例 + live + 树同时开）  
5. 配置面板超长单 scroll，开发者区未折  
6. 主路径中英混排不统一  

### P2 — polish / a11y

7. chip / sheet 键盘与 ARIA  
8. 会话标题截断与次要信息布局  
9. 「进入 →」与整卡点击冗余  
10. dark 下语法高亮与 manifest 主题色  

### 工程隐性税（影响改 UX 的成本，不是用户当面痛点）

11. `app.js` ~4783 行仍扛大部分渲染；`public/js/app/*` 名大于实  
12. 模板字符串 + esc 与 createElement/textContent 混用，一致性一般  

**对照机主既有约束：** 不宜为评价结果擅自大拆 `app.js`；polish 应小步、可测。

## 9. 与设计不变量的对齐检查

| design.md 要求 | UI 是否对齐 | 备注 |
|----------------|------------|------|
| 终端等价（能力不少） | ✅ | 模型/权限/effort/工具/审批在 |
| 审批信息完整性 | ✅（意图+路径） | 完整性预警存在；sheet 语义可再补 |
| 断线重连主路径 | ✅ | 绿点、RTT、续接条、日志 |
| 注意力非对称（触达） | ✅ | 推送相关设置、需要你、提示音震动 |
| 中间层不「优化」Claude 输出 | ✅ | 透传 + 折叠展示，非摘要改写 |
| bypass 显式+危险确认 | ✅（结构） | 高危档有区分样式意图 |
| 不追求 TUI 像素 | ✅ | 聊天形态有意替代 |

**结论：** 产品 UX 与文档北极星**高度同向**；当前短板主要是**表达层 polish**，不是方向错误。

## 10. 横向对比用摘要卡（可直接贴表格）

```
项目: claude-chat-mobile
评价日: 2026-07-16
综合分: 7.5/10

最强三项:
1. 视觉签名（暖米 + serif 阅读态 + Claude 气质 token）
2. 移动端工程细节（safe-area / sheet / 热区 / 键盘 / reduced-motion）
3. 核心工作流与终端等价、审批完整性、「需要你」对齐

最弱三项:
1. 配置模型卡可读性（截图 P0）
2. 空态/状态反馈噪音
3. 无障碍（ARIA/键盘/live region）

最大架构张力:
聊天主面求干净 vs 抽屉/statusline 求 CLI 密度 —— 对 power-user 正确，
应用「默认安静、专家可展开」收敛，而不是删密度。

不建议的「假改进」:
- 弱化权限/effort 入口
- 为好看大拆 app.js
- 引入重型前端框架重做视觉
- 做成大众 ChatGPT 壳

若只改三处立刻提升观感:
1. 模型卡：可读名主标题 + raw id 副标题 + 使用中标记
2. 系统状态条：合并/短消失/别在空态堆三行
3. 附件：缩略图或可区分文件名
```

## 11. 方法说明（对比时防「幻觉分」）

- 截图可核验的问题（模型卡、状态条、附件名、中英混排）标为**高确信度**  
- 代码结构结论来自只读勘察（行数、模块边界、token、a11y 属性抽样），**未**做真机 VoiceOver 实测 → a11y 分为**中确信度**  
- 审批完整命令展示、推送深链等依赖运行时，本批截图未全覆盖 → 记为**设计对齐**而非本轮像素验收  
- 未编造未打开的截图内容；未把「可以更好」说成「产品失败」
