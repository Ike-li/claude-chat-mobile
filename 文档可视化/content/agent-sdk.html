定位

agent.js（931 行）定义 AgentSession 类。每个 AgentSession 对应一个长驻 SDK query() 调用，以 streaming input 模式运行。将 SDK 消息流转化为 agent:event 统一信封，供 server.js 广播至所有前端设备。

核心抽象：一个 AgentSession = 一个打开的 claude 进程。生命周期与前端连接解耦——用户刷新浏览器不会杀死 agent 进程，dispose 才会。

生命周期

  1. constructor()：初始化 instanceId、cwd、permissionMode、effort、epoch、seq 等全部状态。epoch 由 Date.now() + 进程内计数器生成，保证跨重启唯一。
  2. start()：创建 AbortController，调用 SDK query() 传入 inputStream 生成器与 canUseTool 回调。启动 idle 定时器（30s tick）。
  3. consume(q)：消费 SDK query 返回的异步迭代器，每条消息调用 map() 映射为 agent:event。流式事件进入 delta 批处理管道；边界事件直接 emit。
  4. 消息循环：send 将消息推入 queue、触发 inputStream；interrupt 调用 q.interrupt() 并清队列；setPermissionMode 差分调 q.setPermissionMode()。
  5. dispose()：flush delta 缓冲区，清空 toolInputs 缓存，resolve 所有挂起的审批与提问（带 cancelled 语义），清空 bgTasks，调用 abort.abort() 杀进程。

权限闸门流程

  1. handleCanUseTool(name, input, opts) 被 SDK 调用。
  2. AskUserQuestion 特判：先于 bypass 短路之前。问问题与绕过权限审批正交。
  3. dontAsk 防御：SDK 契约保证 dontAsk 不调 canUseTool，此处防御 SDK bug 的误调用。
  4. bypassPermissions：自实现放行（return allow），不依赖 SDK 的 allowDangerouslySkipPermissions。
  5. askPermission：默认路径。emit permission_request → 前端弹出审批弹窗 → 用户决策 → resolvePermission → 返回 allow/deny 给 SDK。

环形缓冲区机制

BUFFER_CAP500 条上限。超限时 shift 丢弃最旧事件。
epochDate.now() + 进程内计数器。跨重启严格递增，客户端据此识别"新流"并重置 seq 去重基线。
seq单调递增序列号。每条 emit 消息获得唯一 seq，emitTransient 不递增 seq。
emit vs emitTransientemit 入 buffer、递增 seq、供回放；emitTransient 不进 buffer、不递增 seq，用于 task_progress 高频心跳。
eventsSince(lastSeq)过滤 buffer 中 seq > lastSeq 的记录。若 bufferTrimmed 且 oldest > lastSeq+1，标记 gap 为 true。

后台任务跟踪

  • bgTaskUpsert(taskId, taskType, message)：心跳刷新。新任务或 taskType 变化回调 onBgTaskChange（重算角标）。
  • bgTaskDone(taskId)：完成信号的精确删除。taskId 为 null 时整表清空（兜底：快任务的心跳 id 未注册进表）。
  • sweepBgTasks()：惰性 TTL 清除。30s 定时器驱动，超 180s 无心跳即判失效清除（BG_TASK_TTL_MS）。
  • hasBgTasks()：判定是否有活跃后台任务。server.js 会话列表据此显示忙碌角标。

Delta 批处理

text_delta 与 thinking_delta 以 20ms 时间窗口 + 2048 字节阈值批量发送，减少 Socket.IO 的 messages/sec 峰值。dispose、interrupt、result 边界均强制 flush 确保不丢失末尾增量。
