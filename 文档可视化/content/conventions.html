分支纪律

分支用途规则
master稳定发布GitHub 默认分支；有分支保护（PR + CI 必过）；git clone 默认拿到 master
dev日常开发所有功能开发在此分支上进行；feature 分支从 dev 切出，PR 合回 dev
feature/*功能开发从 dev 切出；完成后 PR → dev；不在 master 上直接开发

发版流程：dev fast-forward 合并到 master 并打 tag（scripts/release.sh：bump → ff → tag → GitHub Release）。

提交前自检

npm run check              # JS 语法 + 文档一致性 + visual mock registry guard（零 token）
node scripts/doctor.js     # 启动前配置自检
node --test test/*.test.mjs  # 纯逻辑单测（零 token）
npm run test:visual        # puppeteer 移动端视觉回归 E2E

推送至远程前建议执行 CI=true npm test 全量模拟（排除需真 Claude 的集成测试）。

文档极简原则

项目维护少量文档，不设治理层。不写 ADR（架构决策记录），不加 SoT 表/lint 闸/规范文件/数量预算。功能核心文档为 docs/design.md、docs/deployment.md、docs/event-contract.md。规则进入 CLAUDE.md 或保持口头传承。

代码风格

  • 纯 JS ESM，零构建，零打包器
  • console.log + 时间戳足够，不引入"企业级"日志体系
  • 前端 JS 在 public/js/ 下，无框架单页应用
  • 后端模块化拆分在根级 .js 文件中，每模块职责单一

命名约定

instanceId格式 inst_N（N 为整数），进程内唯一、永不重用
sessionId使用 claude CLI 原生 ID，对应 transcript jsonl 文件名
cwd需经 fs.realpathSync 规范化，避免符号链接导致路径不一致
deviceToken由服务端指纹生成；首次连接自动注册，需管理员审批

测试纪律

测试类型命令说明
全量模拟CI=true npm test仅纯逻辑测试，零 token，适合 push 前运行
单元测试node --test test/*.test.mjs仅 test/ 目录下的纯逻辑测试
集成测试npm run test:integration需真 server，位于 test/integration/
可视化 E2Enpm run test:visual零外部依赖 mock server，Puppeteer 驱动
真 Claude 集成RUN_CLAUDE_INTEGRATION=1 npm testopt-in，消耗 token，默认跳过

发版流程

scripts/release.sh 自动化发版：bump 版本号 → fast-forward dev 到 master → 打 tag → 创建 GitHub Release + CI 模拟/ls-remote/trap 护栏。使用 --dry-run 预览操作而不执行。
