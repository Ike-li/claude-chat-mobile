系统架构总览

三层架构、模块全景图、一条消息如何从手机流到 Claude 再回来。

阅读约 14 分钟 第 4 / 27 篇 源自 docs/design.md · docs/interfaces.md

三层架构

系统由三层构成，从前端到后端逐层委托：

1

前端层

部署在浏览器中的单页 PWA。public/index.html 提供应用壳，CSS 自定义属性令牌定义主题色（--canvas:#F4F3EE, --accent:#C96442）、深色模式、Source Serif 4 自托管字体。public/js/app.js（约 3520 行）处理所有 UI 逻辑和 Socket.IO 事件分发，使用三层路由（OOB 旁路 → instanceId 筛选 → epoch/seq 去重）。public/js/sw.js 为 Service Worker，仅用于 Web Push 通知，不做缓存。public/js/logic.js 为纯逻辑函数（零 DOM 零 socket），供单元测试。vendor/ 目录自托管 marked、DOMPurify、highlight.js、Tailwind CSS，不依赖外部 CDN。

2

中间层

Express 5 + Socket.IO 4 服务端（server.js，1766 行）。负责事件路由、鉴权闸门（AUTH_TOKEN → TOFU 设备信赖 → CF Access JWT 三层）、会话编排、推送通知调度、工作区热加载。不存储消息内容，不修改或摘要 Claude 的输出。server.js 编排多 AgentSession 并发，通过 viewingInstanceId 将 agent:event 路由到正确的客户端 tab。

3

后端层

通过 @anthropic-ai/claude-agent-sdk 0.3.201 的 query() 调用本机 Claude Code CLI。agent.js（931 行）的 AgentSession 类封装 SDK 调用，将 SDK 事件映射为统一的 agent:event 信封 {seq, epoch, sessionId, instanceId, cwd, ts, type, payload}，管理权限闸门（handleCanUseTool → askPermission → resolvePermission）、后台任务注册表、500 条环形缓冲区（epoch+seq 组合键去重）。每个 AgentSession 对应一个独立的 claude --resume 子进程。

一条消息的完整旅程

从手机输入到 Claude 回复再到历史回显，一条消息经历 12 步：

  1. 用户在手机浏览器输入消息，点击发送。
  2. app.js 通过 Socket.IO 向服务端 emit "user:message" 事件，附带 sessionId 和 instanceId。
  3. server.js 收到事件，校验 AUTH_TOKEN 和设备信赖状态。未通过则断开连接。
  4. 校验通过后，server.js 将消息路由到对应 instanceId 的 AgentSession。
  5. agent.js 将用户消息包装为 SDKUserMessage，调用 SDK 的 query() 方法。
  6. SDK 将消息通过 stdin 写入本机 Claude Code CLI 子进程。CLI 的 session_id 和 projectDir 由 sessions.js 的 currentByCwd Map 管理。
  7. CLI 调用 Anthropic API，模型开始生成回复。SDK 以事件流形式推送回 agent.js。
  8. agent.js 将每个 SDK 事件映射为统一 agent:event 信封 {seq, epoch, sessionId, instanceId, cwd, ts, type, payload}，写入环形缓冲区。
  9. agent:event 通过 Socket.IO 广播到客户端。app.js 收到事件后通过三层路由（OOB 旁路 → instanceId 筛选 → epoch/seq 去重）分发到对应渲染函数。
  10. 前端增量渲染流式内容。text_delta 追加到消息块，tool_use 渲染为工具卡片，status_line 更新状态栏，permission_request 弹出确认对话框。
  11. CLI 在每次响应完成后将完整 transcript 写入 ~/.claude/projects//.jsonl。
  12. 客户端断线重连后，history.js 读取 JSONL 文件的最新记录，通过 history_append socket 事件回显已错过但已落盘的内容。环形缓冲区提供内存级快速同步。

多实例模型

每个浏览器 tab 对应一个独立的 instanceId。系统维护 instanceId 到 AgentSession 的一对一映射：

instanceId每个 tab 创建时由 server.js 生成（"inst_N"），标记所有出站事件的目标会话。进程内唯一、永不重用。
AgentSession包装 claude --resume 子进程的类。每个 instanceId 独占一个实例，不共享可变状态。
viewingInstanceIdserver.js 用此字段将 agent:event 路由到正确的客户端 tab。前端提交消息时携带此 ID 指定渲染目标。
环形缓冲区每个 AgentSession 维护 500 条最新 agent:event。epoch + seq 组合键去重，防止多 tab 并发时事件重复消费。
cwd每个 session 关联一个工作目录。sessions.js 的 currentByCwd Map 以 cwd 为键索引当前活动 session。

多 tab 共享同一个 AUTH_TOKEN 和设备信赖白名单。浏览器窗口关闭不终止 AgentSession——session 由 server.js 统一管理，直到显式销毁或超时。断线重连后，前端发起 sync:since 请求从指定 epoch 开始增量同步，避免全量重放。

关键设计决策

决策选择理由
通信协议Socket.IO 4内置房间管理（按 instanceId 路由）、自动重连与指数退避、事件命名空间三层路由、WebSocket 降级到 HTTP 长轮询的 fallback 机制。裸 WebSocket 需要自行实现以上全部能力。移动端网络不稳定，自动重连是关键需求。
消息存储不存数据库CLI JSONL 是唯一消息事实源。服务端只存 data/sessions.json（会话元数据映射：cwd → sessionId），永不存消息内容。避免双写一致性问题，消除数据库依赖的运维成本。
会话模型每 tab 一个 AgentSession每个 tab 独立调用 claude --resume。隔离保证多 tab 不共享可变状态。单个 session 崩溃不影响其他 tab。代价是多个 CLI 子进程并发消耗内存。单用户场景下并发数通常不超过 5，内存压力可控。
历史回显session:history socket 事件替代 HTTP GET /history 端点。减少 HTTP 公网端口暴露面，与 Socket.IO 鉴权路径一致，支持增量推送而非全量拉取。
Service Worker 定位仅 Web Push不做缓存策略或离线页面。PWA 核心交互由 app.js 运行时实现，SW 仅承载推送通知通道。降低 SW 复杂度和缓存失效带来的版本问题。
鉴权架构三层 fail-closedAUTH_TOKEN（第一关）→ TOFU 设备信赖（第二关）→ CF Access JWT（第三关，仅公网路）。任何一层校验失败立即断开。不降级、不绕过。每层解决不同维度的暴露风险。
