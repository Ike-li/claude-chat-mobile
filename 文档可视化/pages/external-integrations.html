外部集成

Cloudflare Access JWT、Cloudflare Tunnel、Web Push、ntfy——四个外部系统的对接方式。

阅读约 10 分钟 第 19 / 27 篇 源自 docs/deployment.md · cf-access.js · notifications.js

四个外部系统

Cloudflare Tunnel

命名隧道将 localhost:3000 投递到固定域名。cloudflared 作为独立常驻进程运行（deploy/tunnel.plist.template）。cloudflared tunnel create 创建命名隧道，cloudflared tunnel route dns 绑定 DNS，config.yml 定义 ingress 规则。cloudflared 不参与应用层逻辑，仅做 TCP 隧道。

Cloudflare Access

JWT 验签在应用层完成（cf-access.js）。双层缓存：内存缓存 + 文件缓存（data/cf-access-certs.json），公钥获取超时 8s。fail-closed——JWT 缺失或校验失败直接 401，不降级放行。Access 会话有效期由 Cloudflare 控制，独立于浏览器会话。

Web Push

VAPID 协议实现浏览器推送。Service Worker（sw.js）仅处理推送事件，不做缓存拦截。订阅持久化通过 push/subscribe 端点。推送失败返回 410/404 时自动退订。notificationForEvent 构造推送内容。

ntfy（可选）

自托管 ntfy 服务通过 HTTP POST JSON body 发送。配置 NTFY_URL、NTFY_TOPIC、NTFY_TOKEN 即可启用。消息体支持中文，元数据包括 priority、tags、click 深链 URL。适用于高可靠性推送场景。

PUBLIC_URL 深链机制

PUBLIC_URL 使通知中的链接能复原应用状态。URL hash 参数携带三个上下文字段：instance（目标实例 ID）、session（目标会话 ID）、cwd（目标工作目录，需 realpath 规范化）。客户端 applyDeepLink 解析：instanceId 仍在 live 列表 → setViewing；实例已失效但会话在 → session:switch；都定位不到 → 打开会话列表。
