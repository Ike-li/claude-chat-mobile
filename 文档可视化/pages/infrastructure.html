workdirs.js —— 工作区白名单解析

配置来源：workdirs.json 文件（位于 CCM_DATA_DIR，默认 data/）。支持热加载：server.js 通过 fs.watch 监听该文件变化，变更后重新解析并广播 dirs 更新事件到所有已连接客户端。

白名单解析流程：

  1. 原始输入：JSON 数组，每项可为字符串（路径）或 {path, sessionLimit} 对象。
  2. normalizeWorkdirEntries(parsed)（纯函数）：逐条校验——非数组返回空数组+警告；非法条目 warn-skip；按 path 首见去重；sessionLimit 非法回退 DEFAULT_SESSION_LIMIT（6），超上限夹到 MAX_SESSION_LIMIT（50）。
  3. resolveWorkdirs(entries)：对每项 path 执行 realpathSync 解符号链接/相对段；statSync.isDirectory() 校验。不存在/不可达/不是目录 warn-skip；realpath 后二次去重（首见 sessionLimit 优先）。
  4. 热加载：fs.watch 监听 workdirs.json → 触发重解析 → 广播 dirs 更新（已移除目录上的已开会话继续运行、仅拒新开）。

容错原则：坏配置不清空白名单。loadWorkdirsFile 读/解析失败返回 null，调用方保留旧配置。单个条目标题 warning（不抛错）——server 启动不因某条工作区不可达而中断。

uploads.js —— 附件上传全流程

完整文件字节写入 WORK_DIR/.ccm-uploads/，将绝对路径注入 prompt 文本，claude 通过 Read（白名单内、cwd 内免审批）读取。

  1. 客户端 base64 编码：用户从手机选择文件后，前端读取为 base64 数据，随 user:message 一起发送。
  2. validateAttachments(attachments)（纯校验，零 IO）：上限 10 文件、单文件 10 MiB、总量 20 MiB。
  3. saveAttachments(workDir, attachments)：创建 .ccm-uploads/ 目录；rejectableSymlinkComponent 检查上传目录路径中的可疑符号链接；O_NOFOLLOW + O_EXCL 标志打开文件（防 TOCTOU + symlink 攻击）；路径校验防穿越。
  4. buildPromptText(text, saved)：在原文末尾追加 [附件] 段（绝对路径逐行列出）。
  5. toEventMeta(saved)：为 user_message 事件提供元数据（剥掉 absPath 与完整 data，仅保留 name/mimeType/size/thumb——不泄露服务端路径）。

devices.js —— TOFU 信任模型

每台首次连接的浏览器生成随机 16 字节指纹（deviceToken），存入 localStorage.device_token。服务端双文件持久化：

data/trusted-devices.json已批准设备 token 的 Set，序列化为数组。
data/pending-devices.json待审批设备列表，每项含 {deviceToken, ip, userAgent, ts}。

生命周期：首次连接 → addPendingDevice → 管理员在任一客户端审批或拒绝 → approveDevice 将 token 加入 trusted set 并移出 pending。每次 isDeviceTrusted 调用都会重新 loadTrustedDevices()，确保多进程/CLI 操作能即时感知。

statusline.js —— Web 状态栏

数据来源：SDK lastUsage（input_tokens/cache tokens 计数）、git status（execFile 获取 branch/changed/ahead/behind）、CLI 版本号（启动时采集）。组装逻辑：task（当前子任务名）、model（activeModel/reportedModel）、cwd/project、git（branch + 变更数 + repo）、ctx（token 明细 + 缓存命中率）、cost（累计 USD + wall/API 时间）、version（CLI 版本号）。缓存策略：gitCache 针对每个 cwd 缓存 5s；cache 失效倒计时是客户端推算（CACHE_TTL_MS=300000），非 SDK 回报的权威值。

notifications.js —— 通知双渠道

事件类型文案模板抑制条件ntfy 优先级
result"任务完成/出错" + 耗时有客户端连接时抑制3（default）
permission_request"Claude 请求许可" + 工具名+参数摘要无条件推送（可能锁屏）5（urgent）
question"Claude 有问题" + 文本摘录无条件推送5（urgent）
task_notification"后台任务完成/失败" + 摘要无条件推送3（default）

双渠道：Web Push（Service Worker push 事件 + showNotification）与 ntfy（自托管 ntfy server 的 POST 请求）。ntfy 用 JSON body（非 HTTP header）传递标题，避开中文标题的 header 编码问题。消息体中的 instanceId/sessionId/cwd 用于生成深链 click URL。

models-cache.js —— 按 cwd 缓存的模型清单

模型清单来自 SDK supportedModels()，而工作区的 .claude/settings.local.json 的 env 块可覆盖 ANTHROPIC_BASE_URL 与自定义模型名。因此模型清单是随工作区变动的量，绝非账号级全局。单全局缓存会把上个工作区的模型泄漏进当前工作区视图。

实现：createModelsCache({max=32}) 返回 {set, get, toJSON, load}。LRU-on-write 淘汰。get 对未知 cwd 返回 null，绝不回退别的 cwd 的清单。toJSON/load 支持跨重启持久化到 data/init-cache.json（缓存可弃，损坏即当空）。isCwdDefaultModel 守卫：仅当 resumeId==null 且 pinnedModel===undefined 且 reportedModel 非空时，才将 init.model 视为 cwd 默认模型并入缓存。
